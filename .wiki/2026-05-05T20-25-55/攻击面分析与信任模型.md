# 攻击面分析与信任模型

## 信任模型概述

该系统的**信任最小化**建立在三层约束之上：**密码学约束**（drand 多签名不可伪造）、**逻辑约束**（参数在 URL/短码中锁定、算法确定性）、**社会约束**（博主必须在截止前公开参数、粉丝需核对名单）。前两层在代码层面强制执行，第三层依赖社区监督。理解这三层的边界——什么被代码保护、什么需要社群介入——是正确使用本系统的前提。

[来源](PLAN.md#L16-L19)，[来源](GUIDE.md#L211-L228)

## 五种攻击场景与防御机制

### 1. 博主选择 Round：deadline 锁定 round

**攻击**：博主在获知 drand 随机数后，回溯选择一个对自己有利的 round 号，声称该 round 才是抽奖所用。

**防御**：Round 不是由博主选择的，而是由 **deadline（截止时间戳）** 通过公式确定性计算得出：

```js
round = Math.floor((deadline - genesisTime) / period) + 1
```

deadline 在创建抽奖时被编码进 URL Hash 参数和短码中，一旦公开即不可更改。页面从链接或短码中解码出 deadline，自动计算出对应 round，博主无法对不同的验证者宣称不同的 round 号。粉丝只需打开链接即可看到预期 round 号，博主无从抵赖。

[来源](src/lottery.js#L4-L9)，[来源](src/encode.js#L5-L18)

### 2. 虚报 N：N 在 URL/短码锁定

**攻击**：博主对 A 说「参与者共 100 人，你是 #42 中奖」，对 B 说「参与者共 200 人，你是 #42 中奖」，利用不同 N 值让同一编号对不同的人看起来都是中奖。

**防御**：参与人数 N 同样编码在 URL/短码中，页面通过 `decodeShortCode` 或 URL 参数解析后直接渲染显示，博主无法对不同的粉丝展示不同的 N。任何人对链接或短码的解析结果完全一致，不存在信息差。

[来源](src/encode.js#L12-L13)，[来源](src/encode.js#L29-L31)

### 3. 篡改排序规则：排序规则硬编码

**攻击**：博主使用不同的排序规则（如按时间倒序、按字母倒序、随机排序）来操控谁出现在哪个编号位置，从而让特定参与者中奖。

**防御**：排序规则由页面**硬编码声明**，写在 `SortTool.js` 的 `parseHandles` 函数中：

```js
handles.sort((a, b) => a < b ? -1 : a > b ? 1 : 0)   // ASCII 升序
```

规则在 [ALGORITHM.md](ALGORITHM.md) 中明确文档化：「按 ASCII 码升序排列—— `"0" < "9" < "@" < "A" < "Z" < "_" < "a" < "z"`」，粉丝可自行核对第 0 号是否确实是 ASCII 排序后的第一个 handle。排序工具生成的可导出图片也使用一致的排序算法，博主无法声称图片使用的是另一套规则。

[来源](src/components/SortTool.js#L163-L166)，[来源](ALGORITHM.md#L22-L27)

### 4. 伪造随机数：drand 多签名验证

**攻击**：博主伪造一个自己生成的随机数代替 drand 的真实输出，操控中奖结果。

**防御**：drand 响应中包含 `randomness` 和 `signature` 两个字段。`signature` 由 League of Entropy 的分布式节点使用 BLS 门限签名生成，任何人均可通过 drand 网络公开的 `publicKey` 验证签名的正确性。页面从 drand 官方 relay（`api.drand.sh`、`api2.drand.sh`、`drand.cloudflare.com`）直接获取数据，采用故障切换机制逐一尝试，博主无法篡改传输过程。粉丝也可以通过 `drand-client`（drand 官方 SDK）或 curl 请求独立验证同一 round 的 `randomness` 是否与页面显示一致。

[来源](src/api.js#L1-L18)，[来源](src/chains.js#L1-L42)

### 5. 多次开奖：确定性计算

**攻击**：博主反复开奖直到对结果满意，将某次有利的结果公布给粉丝。

**防御**：中奖计算是**完全确定性的纯函数**。给定相同的输入 `(randomness, N, prizeTiers)`，`computeWinners` 始终返回完全相同的输出。drand 的 round 由 deadline 唯一确定，因此对一个给定的 URL/短码，**只有一个可能的中奖结果**。博主无法多次重算，因为输入已经被锁定。粉丝可以自己从 drand 获取 `randomness`，运行相同的 `computeWinners` 验证结果，如果与博主声称的不一致则说明博主造假。

[来源](src/lottery.js#L12-L36)，[来源](PLAN.md#L27-L32)

## 仍需社会约束的三个方面

以上五种攻击在代码层面被完全阻断。但以下三点在当前的纯前端架构中无法用代码强制约束，需要**社群监督和博主诚信**：

### 截止前公开链接

如果博主在截止时间之后才公布链接/短码，旁观者无法证明 deadline 在创建时就是现在的值——博主可能在截止后创建一个新的、对已知道的 drand round 更有利的 deadline。**解决方式**：博主应在发布抽奖活动时就附上链接或短码，粉丝此时即可查看截止时间、N、奖项配置，并自行设置日历提醒。页面在截止前显示倒计时，粉丝可截图留证。

相关流程见 [博主操作指南](博主操作指南.md)。

[来源](GUIDE.md#L219)

### 开奖前公开完整候选列表

系统无法知道真实的参与者名单——页面只存储 N（人数），不存储具体是谁。如果博主在开奖后调整名单（例如将不中奖的用户移除、让编号前移），粉丝无法察觉。**解决方式**：博主应在截止时（Phase 2，尚未开奖）公开完整的、已排序编号的候选列表。粉丝需自行核对自己的 handle 是否在列表上、编号是否正确。系统的 [候选列表排序与导出](候选列表排序与导出.md) 工具内置了 ASCII 排序和图片导出功能，方便博主截屏公示。

[来源](GUIDE.md#L220)

### 诚信去重

同一用户多次点赞/评论/转发仅应计为一次参与。系统在 `parseHandles` 中确实有内置去重逻辑（`[...new Set(handles)]`），但这只能去除**完全相同的字符串**。对于同一个用户使用不同设备、不同账号互动的情况，系统无法自动识别。**解决方式**：博主在整理候选列表时需自行做跨账号/跨平台的去重，并在公开列表前说明去重规则。粉丝看到名单中有重复时有权质疑。

[来源](src/components/SortTool.js#L157-L158)

## 信任模型总结

| 信任层 | 保证内容 | 强制执行方 | 可验证方式 |
|--------|----------|------------|------------|
| **密码学** | drand 随机数不可预测、不可伪造 | League of Entropy（BLS 门限签名） | 用 drand publicKey 验签 |
| **逻辑** | Round 锁定、N 锁定、算法确定 | 代码（URL/短码编码 + 纯函数） | 打开链接/短码核对 + 本地重算 |
| **社会** | 截止前公开、名单完整、诚信去重 | 社群监督 + 博主声誉 | 截图留证 + 核对已公开名单 |

前两层提供了密码学级别的保障，第三层需要所有参与者的共同维护。**信任不是全有或全无，而是分层叠加的**：即使博主不诚信，他的造假行为也是公开可见、可被任何人独立证伪的，这就是「可验证」的核心含义。
