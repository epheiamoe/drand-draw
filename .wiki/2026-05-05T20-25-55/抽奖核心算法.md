# 抽奖核心算法

整个系统的信任根基建立在一行公式上：`round = floor((deadline - genesis) / period) + 1`。这条公式将博主的**截止时间**绑定到 drand 去中心化随机信标网络的**唯一一个 Round 编号**上，再从这个 Round 的 randomness 派生出每一个中奖编号。算法全流程分五步，每一步都是确定性的——输入相同，输出必相同。

## 1. Round 计算：截止时间 → 链上轮次

drand 网络以固定间隔（period 秒）产生一个随机数，每个轮次有唯一的 round 编号。抽奖的 round 由截止时间唯一确定：

```
round = floor((deadlineUnix - genesisTime) / period) + 1
```

### 链配置参数

| 链 | genesisTime | period | genesisTime 含义 |
|---|---|---|---|
| quicknet | 1,692,803,367 | 3s | 2023-08-23 08:29:27 UTC |
| default | 1,595,431,050 | 30s | 2020-07-22 08:17:30 UTC |
| evmnet | 1,727,521,075 | 3s | 2024-09-28 10:57:55 UTC |

### 实现细节

```js
// src/lottery.js
export function computeRound(chainId, deadlineUnix) {
  const chain = CHAINS[chainId]
  const elapsed = deadlineUnix - chain.genesisTime
  if (elapsed < 0) return -1      // 截止时间早于创世时间 → 无效
  return Math.floor(elapsed / chain.period) + 1
}
```

```python
# cli/drand_draw/lottery.py
def compute_round(deadline, genesis, period):
    elapsed = deadline - genesis
    if elapsed < 0:
        return -1
    return math.floor(elapsed / period) + 1
```

关键推理：`+ 1` 是因为 drand 的 round 编号从 1 开始，而非从 0。`elapsed / period` 得到的是截止时间之前已经完成的完整轮次数，再加 1 才是截止时刻之后的**下一轮**——即抽奖将要使用的随机数所在的轮次。

## 3. SHA-256 种子派生：randomness + shift → 256 位哈希

核心种子派生公式：

```
seedHex = SHA256(randomness + ':' + shift)
```

其中 `shift` 是当前已确定的中奖编号数量（即 `winners.length`），从 0 开始递增。每个奖位的 shift 不同，因此每个奖位的种子也完全不同。

### 为什么用 SHA-256 而非直接拼接？

这是系统早期版本经历过的一次重要升级。最初的实现是直接拼接：

```
seedHex = randomness + shift.toString(16)   // 旧版，已废弃
```

这个做法有一个严重问题：当 shift 很小（如 0、1、2、3）时，seedHex 的前缀完全相同（都是 randomness），只有最后几位不同。对于一个 64 字符的十六进制数来说，最后几位的改变对 BigInt 取模结果的影响极其有限，**连续多个奖位的取模结果高度相关，极易产生连续编号**。

SHA-256 解决了这个问题：

- **雪崩效应**：输入中任意一个比特的变化，输出中大约 50% 的比特会翻转。`randomness + ":0"` 和 `randomness + ":1"` 的哈希结果在视觉上毫无关联。
- **独立性保证**：每个 shift 对应一个在密码学意义上独立的伪随机数，多奖项之间的取模结果互不相关。
- **安全归约**：攻击者无法通过已知的某个奖位编号反推出其他奖位。

这次变更记录在 commit `62b2ec8` 中，标题为 _"Fix: use SHA-256 for winner derivation (prevents consecutive winners)"_。

## 4. BigInt 取模：256 位哈希 → 编号空间

将 64 字符的十六进制哈希视为一个大整数，对 N（参与人数）取模：

```js
const bigVal = BigInt('0x' + seedHex)   // 256 位整数
let idx = Number(bigVal % BigInt(n))     // 映射到 [0, N-1]
```

为什么用 BigInt？SHA-256 输出 256 位，远超 JavaScript Number 的安全整数范围（53 位）。Python 的 `int` 原生支持任意精度，因此 `int(seed_hex, 16) % n` 就直接可用。

```
seedHex = "e7b2a9d1...f03c"      // 64 字符十六进制
bigVal  = 0xe7b2a9d1...f03c      // ≈ 2^255 量级的大整数
idx     = bigVal % 100            // 映射到 0-99
```

## 5. 碰撞处理：环形顺延

当取模结果 `idx` 已被前面的奖位占用时，执行环形线性探测（circular linear probing）：

```js
while (used.has(idx)) {
  idx = (idx + 1) % n
  attempts++
  if (attempts >= n) throw new Error('Collision resolution failed')
}
used.add(idx)
winners.push(idx)
```

`attempts >= n` 的兜底检查理论上不可能触发——只有在所有 N 个编号全部被占用时才会发生，但算法在进入循环前已经验证了 `totalPrizes <= N`，所以一定有空位。

### 完整的 `computeWinners`

将上述步骤整合为完整的算法：

```js
export async function computeWinners(randomness, n, prizeTiers) {
  const totalPrizes = prizeTiers.reduce((a, b) => a + b, 0)
  if (totalPrizes > n) {
    throw new Error(`Total prizes (${totalPrizes}) exceeds N (${n})`)
  }
  const winners = []
  const used = new Set()
  for (let i = 0; i < prizeTiers.length; i++) {
    for (let j = 0; j < prizeTiers[i]; j++) {
      const shift = winners.length
      const seedHex = await deriveSeed(randomness, shift)
      const bigVal = BigInt('0x' + seedHex)
      let idx = Number(bigVal % BigInt(n))
      let attempts = 0
      while (used.has(idx)) {
        idx = (idx + 1) % n
        attempts++
        if (attempts >= n) throw new Error('Collision resolution failed')
      }
      used.add(idx)
      winners.push(idx)
    }
  }
  return winners
}
```

Python 实现结构完全一致，区别仅在于 `int(seed_hex, 16)` 替代 `BigInt('0x' + seedHex)`，以及 `hashlib.sha256()` 替代 `crypto.subtle.digest('SHA-256')`。[算法规范与测试向量](算法规范与测试向量.md) 页面提供了跨平台一致性验证的具体方法。

## 手动计算示例

假设：
- randomness = `"a3f25c8d1e7b..."`（64 字符，此处简写）
- N = 100
- prizeTiers = [1, 3]（一等奖 1 人，二等奖 3 人）

| 奖项 | shift | seedHex（SHA-256 输出） | bigVal % 100 | 碰撞？ | 最终编号 |
|---|---|---|---|---|---|
| 一等奖 | 0 | `e7b2a9d1...` | 42 | 否 | **#42** |
| 二等奖(1) | 1 | `8d3f1c2b...` | 15 | 否 | **#15** |
| 二等奖(2) | 2 | `5a9e8f7c...` | 78 | 否 | **#78** |
| 二等奖(3) | 3 | `f03c2a1b...` | 42 | 是（#42 已用）→ 顺延 43 | **#43** |

第三个二等奖位产生碰撞（42 已被一等奖占用），环形顺延到 43——43 未被占用，所以三等奖得主为 #43。

注意如果没有 SHA-256，shift=0 到 shift=3 的 seedHex 会是 `a3f25c...0`、`a3f25c...1`、`a3f25c...2`、`a3f25c...3`——前 62 个字符完全相同，取模结果几乎必然集中在相近的编号区间。

## 多奖项分层逻辑

`prizeTiers` 是一个数组，如 `[1, 3]` 表示"一等奖 1 人，二等奖 3 人"。算法按数组顺序逐层逐个奖位计算，`formatWinnerList` 函数将扁平的 `winners` 数组重新按层级分组：

```js
export function formatWinnerList(winners, prizeTiers) {
  const result = []
  let idx = 0
  for (let i = 0; i < prizeTiers.length; i++) {
    const tierWinners = winners.slice(idx, idx + prizeTiers[i])
    result.push(tierWinners)
    idx += prizeTiers[i]
  }
  return result
}
```

分组结果：`[[42], [15, 78, 43]]` 对应一等奖 [#42]，二等奖 [#15, #78, #43]。

## 设计总结

整个算法是**确定性、透明、无可争议**的。博主在开奖前无法预知结果（因为 randomness 来自未来时刻的 drand 轮次），开奖后任何人都可以用相同的参数复现完全相同的中奖编号。系统不依赖任何后端的诚实——信任由算法和密码学保证，而非中心化权威。

算法的跨平台一致性（JS 浏览器端 vs. Python CLI）通过共享的测试向量验证，详情见 [算法规范与测试向量](算法规范与测试向量.md)。完整的算法规范文档见项目根目录的 `ALGORITHM.md`。
