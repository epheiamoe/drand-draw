# 短码编解码规范

短码是整个系统中信息交换的**核心序列化格式**。它将一次抽奖的全部参数（链、截止时间、参与人数、奖项配置、中奖结果）编码为一个紧凑的、可读的、社交平台友好的字符串。短码与 URL Hash 参数是等价表示——两者之间可无损互转，且都服务于同一目的：让抽奖参数在无后端存储的条件下可靠传递。

## 格式总览

短码由若干 **`-` 分隔的字段** 组成，其正则模式为：

```
^[qde]-[0-9a-f]+-[0-9a-z]+(?:-[0-9a-z,]+)*(?:-[0-9a-z,]+)*$
```

具体分为 5 个字段，其中后两个为可选：

| 位置 | 字段 | 必需 | 编码方式 | 示例 |
|------|------|------|----------|------|
| 1 | 链缩写 | 是 | 单字母 `q`/`d`/`e` | `q` |
| 2 | 截止时间戳 | 是 | 小写十六进制（无 `0x` 前缀） | `66364280` |
| 3 | 参与人数 N | 是 | base36（`0-9a-z`） | `2s` |
| 4 | 奖项配置 | 否 | 逗号分隔的 base36 数字串 | `1,3` |
| 5 | 中奖编号 | 否 | 逗号分隔的 base36 数字串 | `2a,f,4e` |

完整示例：

```
q-66364280-2s-1,3-2a,f,4e
```

## 字段详解

### 字段 1：链缩写（`q` / `d` / `e`）

三条支持的 [drand 链](链配置系统.md) 被映射为三个单字母缩写：

| 缩写 | 链 ID | genesis | period | 典型用途 |
|------|-------|---------|--------|----------|
| `q` | quicknet | 1692803367 | 3s | 快速验证、演示 |
| `d` | default | 1595431050 | 30s | 常规抽奖 |
| `e` | evmnet | 1727521075 | 3s | 与 EVM 生态对齐 |

映射关系在 `chains.js` 和 `encode.py` 中以字典形式对称声明：

```js
// src/chains.js
export const CHAIN_SHORT = { q: 'quicknet', d: 'default', e: 'evmnet' }
export const SHORT_CHAIN = { quicknet: 'q', default: 'd', evmnet: 'e' }
```

解码时，遇到未知前缀默认回退 `quicknet`——这不是容错设计，而是为了向后兼容早期未规范化前缀的短码。

[来源](src/chains.js#L54-L55) | [来源](cli/drand_draw/encode.py#L1-L19)

**设计意图**：
- 三字母缩写（如 `q` vs `qui`）将链信息的固定开销降至 **1 字符**。
- 选择 `q/d/e` 而非 `0/1/2` 的原因是字母可以自然关联链名的首字母（quicknet → **q**，default → **d**...evmnet → **e**），降低记忆负担。

---

### 字段 2：截止时间戳（小端十六进制）

**deadline** 是 Unix 时间戳（自 1970-01-01 UTC 起的秒数），编码为小写十六进制字符串。

```js
// src/encode.js
parts.push(params.deadline.toString(16))
```

```py
# cli/drand_draw/encode.py
deadline_hex = format(deadline, 'x')
```

示例：时间戳 `1715000000` → `66364280`（8 个十六进制字符）。

**设计意图**：
- 十六进制编码将 10 位十进制数字压缩为 **7-8 位**（最多到 8 位，因为 32 位时间戳最大值 `0xFFFFFFFF` = 4294967295 对应 2106 年，足够覆盖所有现实场景）。
- 人类可直接查看十六进制数值是否有明显异常（如全零、超长），作为快速校验。
- 小写字母与 base36 字段在视觉上统一。

[来源](src/encode.js#L6) | [来源](cli/drand_draw/encode.py#L43)

### 字段 3：参与人数 N（base36）

N 是参与总人数，编码为 **base36**（字符集：`0-9a-z`）。

```js
// src/encode.js
parts.push(params.n.toString(36))
```

```py
# cli/drand_draw/encode.py
def to_base36(n):
    if n == 0:
        return '0'
    result = []
    while n > 0:
        result.append(_BASE36_CHARS[n % 36])
        n //= 36
    return ''.join(reversed(result))
```

**为什么是 base36 而不是 base10 或 base64？**

- **对比 base10**：base36 的紧凑度更高。一个 6 位数（如 `100000`）在 base10 是 6 字符，base36 仅需 4 字符（`255s`）。N 上限为 10 万级时，base36 可节省约 30% 的空间。
- **对比 base64**：base64 虽更紧凑，但引入了 `+`、`/`、`=` 等符号，这些符号在 URL 中需要转义，且与字段分隔符 `-` 和列表分隔符 `,` 竞争视觉层级。base36 纯字母数字，无需任何转义。
- **大小写敏感性**：统一使用小写，避免大小写混淆（如 `O` vs `0`）。

### 字段 4：奖项配置（base36 逗号分隔）

每个奖项层级的人数，按顺序以逗号分隔，每个值均为 base36。

```js
// src/encode.js
parts.push(params.prizes.map(p => p.toString(36)).join(','))
```

例如配置"一等奖 1 人、二等奖 3 人" → `1,3`。

这一字段的存在/缺失标志着抽奖是否已配置奖项。若缺失，解码为 `[]`。

**设计意图**：
- 逗号作为列表分隔符，是 CSV 惯例，阅读友好。base36 保证每个元素不含逗号，因此分隔符无需转义。
- 奖项**顺序**即层级顺序，隐含了一等奖在前的语义。

[来源](src/encode.js#L8-L10) | [来源](cli/drand_draw/encode.py#L46-L47)

---

### 字段 5：中奖编号（base36 逗号分隔）

抽奖完成后，每个中奖者在排序列表中的编号（从 0 开始），以逗号分隔的 base36 数字串呈现。

```js
// src/encode.js
parts.push(params.winners.map(w => w.toString(36)).join(','))
```

示例：中奖编号 `[42, 15, 78, 33]` → `2a,f,4e,21`。

**重要语义**：
- 中奖编号是**扁平的**——如果奖项配置为 `1,3`（共 4 个奖位），winners 就是一个 4 元素列表，前 1 个为一等奖，后 3 个为二等奖。
- 解码后的 winners 和 prizes 通过数量对应：`prizes[0]` 个元素属于第一级奖项。
- 若 winners 字段缺失，解码为 `[]`，表示尚未开奖。

## 编解码流程

```
编码方向（params → short code）:
  {chain, deadline, n, prizes[], winners[]}
      → [prefix, deadline_hex, n_base36, prizes_base36_csv, winners_base36_csv]
      → join('-')

解码方向（short code → params）:
  "q-66364280-2s-1,3-2a,f,4e"
      → split('-')
      → parts[0] → CHAIN_SHORT → chain
      → parts[1] → parseInt(16) → deadline
      → parts[2] → parseInt(36) → n
      → parts[3] → split(',').map(parseInt(36)) → prizes[]
      → parts[4] → split(',').map(parseInt(36)) → winners[]
```

两个平台实现此流程的代码几乎逐行对应：

| 步骤 | JavaScript (`src/encode.js`) | Python (`cli/drand_draw/encode.py`) |
|------|-----------------------------|--------------------------------------|
| 编码入口 | `encodeShortCode(params)` | `encode_short_code(chain, deadline, n, prizes, winners)` |
| 链前缀 | `SHORT_CHAIN[params.chain]` | `CHAINS[chain_name]['prefix']` |
| deadline | `params.deadline.toString(16)` | `format(deadline, 'x')` |
| N | `params.n.toString(36)` | `to_base36(n)`（手动实现） |
| 分隔符 | `parts.join('-')` | `'-'.join(parts)` |
| 解码入口 | `decodeShortCode(code)` | `decode_short_code(code)` |
| 链还原 | `CHAIN_SHORT[parts[0]]` | `PREFIX_TO_CHAIN[prefix]` |
| deadline 解析 | `parseInt(parts[1], 16)` | `int(parts[1], 16)` |
| N 解析 | `parseInt(parts[2], 36)` | `from_base36(parts[2])` 即 `int(s, 36)` |

两个实现在每个编解码步骤上完全一致，确保同一短码在浏览器和 CLI 中产生相同的抽奖参数。

## 完整编解码示例

以下示例使用 `q-66364280-2s-1,3-2a,f,4e` 演示完整往返：

### 编码（参数 → 短码）

| 参数 | 值 | 编码后 |
|------|-----|--------|
| chain | `quicknet` | `q` |
| deadline | `1715000000` | `66364280` |
| n | `100` | `2s` |
| prizes | `[1, 3]` | `1,3` |
| winners | `[42, 15, 78, 33]` | `2a,f,4e,21` |
| **短码** | | **`q-66364280-2s-1,3-2a,f,4e,21`** |

### 解码（短码 → 参数）

```
"q-66364280-2s-1,3-2a,f,4e,21"
  → prefix 'q'        → chain: 'quicknet'
  → '66364280' (16)   → deadline: 1715000000
  → '2s' (36)         → n: 100
  → '1,3' (36 each)   → prizes: [1, 3]
  → '2a,f,4e,21' (36) → winners: [42, 15, 78, 33]
```

Prizes 和 winners 之间的层级对应关系：`prizes[0]` = 1 对应一等奖 1 人（winners[0] = 42），`prizes[1]` = 3 对应二等奖 3 人（winners[1..3] = [15, 78, 33]）。

### 边界情况测试

| 场景 | 短码 | 解码验证 |
|------|------|---------|
| 最小有效 | `q-0-0` | chain=quicknet, deadline=0, n=0, prizes=[], winners=[] |
| 无奖项 | `q-66364280-2s` | prizes=[], winners=[] |
| 仅 prizes | `q-66364280-2s-1,3` | prizes=[1,3], winners=[] |
| 完整 | `q-66364280-2s-1,3-2a,f,4e,21` | prizes=[1,3], winners=[42,15,78,33] |
| N = 0 | `q-66364280-0-1,3` | n=0 在算法层面是无效的（无参与者），但编码层不拒绝 |

解码器的容错行为：无效的 base36 或 hex 值被 `filter(x => !isNaN(x))` 静默移除。空字段（如 `parts[3]` 为空串）被跳过。

## 设计决策总结

| 决策 | 选型 | 拒绝方案 | 原因 |
|------|------|----------|------|
| 数值编码 | base36 | base10、base64 | 比 base10 紧凑；比 base64 避免特殊字符 |
| 链标识 | 单字母 `q/d/e` | 全名 `quicknet` | 固定开销仅 1 字符，首字母关联便于记忆 |
| 字段分隔符 | `-` | `/` `_` `.` | `-` 在 URL path 和文本中不易歧义，视觉清晰 |
| 列表分隔符 | `,` | `+` `\|` | CSV 惯例，阅读友好 |
| 时间戳编码 | 小端十六进制 | 十进制 | 比十进制缩减约 20% 长度 |
| 大小写 | 全小写 | 混合或大写 | 避免 `O/0`、`I/1` 等视觉混淆 |

## 与链接格式的关系

短码与 [前端路由](前端路由与状态管理.md) 中的 Hash 参数格式通过 `paramsToHash()` / `hashToParams()` 互转：

- **Hash 参数**：`#/?chain=quicknet&deadline=1715000000&n=100&prizes=1,3&winners=42,15,78,33`
- **短码**：`q-66364280-2s-1,3-2a,f,4e,21`

两者携带完全相同的信息，短码仅为社交平台（如 X/Twitter）对外链限流的规避手段。`smartParse()`（详见[智能解析引擎](智能解析引擎.md)）可自动识别输入是短码、URL、URL 片段还是原始键值对，无需用户手动区分。
