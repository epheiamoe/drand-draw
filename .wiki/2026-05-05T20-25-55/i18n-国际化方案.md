# i18n 国际化方案

整个前端界面采用 **运行时双语切换** 策略，不依赖编译期多语言打包工具，不引入 i18n 框架。国际化实现在一个 **58KB 的单文件**（含所有翻译文本）中，通过两套机制协作完成：语言自动检测 + 全量替换式渲染。

## 语言检测：从浏览器到持久化

用户首次访问时，`detectLang()` 决定显示哪种语言。它的决策链只有两级：

```
localStorage 优先 → 浏览器语言回退
```

```javascript
export function detectLang() {
  const stored = localStorage.getItem('drand-lang')
  if (stored && LANGUAGES.includes(stored)) return stored
  const nav = (navigator.language || '').toLowerCase()
  return nav.startsWith('zh') ? 'zh' : 'en'
}
```

第一级查 `localStorage` 中的 `drand-lang` 键——只要用户曾手动切换过语言，这个值就会被写入，此后每次访问都沿用该偏好。第二级是回退策略：读取 `navigator.language`，提取语言标记的前两位，若以 `zh` 开头则返回中文，否则默认英文。

这种"存储优先、浏览器回退"的设计实现了两个目的：**尊重用户主动选择**，同时**首次访问零配置自适应**。

## T 对象：翻译数据的双层结构

所有翻译文本集中定义在导出对象 `T` 中，按语言分为两层：

```
T
├── zh        ← 中文（默认语言，用作全面回退）
│   ├── siteTitle: '可验证抽奖'
│   ├── prizeTier: (i) => `第${'一二三四五六七八九十'[i]||(i+1)}等奖`
│   └── ...
└── en        ← 英文
    ├── siteTitle: 'Verifiable Lottery'
    ├── prizeTier: (i) => `Tier ${i+1}`
    └── ...
```

两层结构遵循一个关键约束：**每个键在两语言中都必须存在**。缺少某个键时，`t()` 会回退到 `T.zh` 中的对应值，因此中文相当于兜底语言。

[来源](src/i18n.js#L12-L132)

### 函数类型值

部分翻译值不是普通字符串，而是 **函数**。最典型的例子是 `prizeTier`：

```javascript
prizeTier: (i) => `第${'一二三四五六七八九十'[i]||(i+1)}等奖`
```

设计意图很清晰：奖项层级的数量（一等奖、二等奖……）在创建抽奖时动态决定，无法用静态字符串穷举。函数接收索引参数 `i`，返回对应层级的名称。中文版本甚至利用汉字数字实现了**超出十等奖的优雅降级**——`'一二三四五六七八九十'[i]` 覆盖前十个奖项，超出时回退到阿拉伯数字。

另一个函数类型是 `prizeTierEn`（英文奖项名），但英文版本直接使用 `Tier ${i+1}` 格式，语义上更接近静态模板而非语言特化——它存在的意义是允许调用方按需选择中英文格式，灵活性大于 `prizeTier`。

[来源](src/i18n.js#L26-L27)

## 两个查找函数：t() 与 tLang()

`t()` 和 `tLang()` 功能重叠但适用场景不同：

| 函数 | 参数 | 查询语言源 | 典型场景 |
|------|------|-----------|---------|
| `t(key, ...args)` | 翻译键 + 函数参数 | 当前语言 `currentLang` | UI 模板中的文本渲染 |
| `tLang(lang, key, ...args)` | 语言代码 + 翻译键 + 函数参数 | 显式指定的语言 | 需要在特定语言上下文中取值 |

两者实现几乎一致——从语言层取值，若为函数则展开调用，缺失时回退到 `T.zh`：

```javascript
export function t(key, ...args) {
  const val = T[currentLang]?.[key] || T.zh[key]
  return typeof val === 'function' ? val(...args) : val
}
```

`t()` 是主力函数，所有 UI 组件通过它获取翻译文本。`tLang()` 则用在 **需要跨语言获取文本** 的地方——例如 `SortTool` 中同时需要中英文标题时：

```javascript
// SortTool 中根据当前语言显示半中半英的描述
${lang === 'en' ? 'Participant List Sorter' : '候选列表排序工具'}
```

注意 `SortTool` 并未使用 `tLang()`，而是直接读取 `getLang()` 后用三元表达式判断——这说明 `tLang()` 在实际代码中的使用频率很低，**绝大多数场景只需要 `t()` 就够了**。

## 语言切换：全页重渲染

语言切换按钮的渲染和绑定在 `main.js` 的 `render()` 函数中：

```javascript
app.querySelector('#lang-switch').addEventListener('click', () => {
  const newLang = getLang() === 'zh' ? 'en' : 'zh'
  setLang(newLang)
  render()
})
```

`setLang()` 负责三件事：

1. 更新内存变量 `currentLang`
2. 写入 `localStorage` 实现持久化
3. 设置 `<html>` 标签的 `lang` 属性（`zh-CN` / `en`），对无障碍和搜索引擎友好

切换触发 `render()` 后，整个 `#app` 的 `innerHTML` 被重新赋值，所有 `t()` 调用在渲染时重新执行，取出新语言的对应文本。

**为什么是全量重渲染而非局部更新？** 因为整个应用是纯函数式渲染模型——`render()` 根据当前状态（路径、语言、抽奖参数）生成完整的 DOM 字符串，不存在"diff 更新"机制。语言切换只是改变了全局状态中的一个变量，然后走了一遍和页面加载时完全相同的渲染路径。

这种模式的代价是切换语言时会有短暂的闪烁（整个页面替换），但好处是逻辑极其简单——**理解和维护成本几乎为零**。

## Guide 组件的双语

`Guide` 组件（使用说明页）的语言处理方式与其他组件不同。它不依赖 `t()`，而是根据 `getLang()` 加载不同的 **外部 Markdown 文件**：

```javascript
const mdFile = lang === 'en' ? '/GUIDE_EN.md' : '/GUIDE.md'
const md = getCachedMd(lang)
```

这是因为使用说明的内容远多于 UI 文本，嵌入到 `T` 对象中会使其膨胀到不可维护。Markdown 文件由 CI 或构建流程管理，应用在运行时只负责按语言加载和渲染。

[来源](src/components/Guide.js#L5-L7)

## 如何添加新语言

在现有架构下添加一种新语言（如日语 ja）只需四步：

1. **在 `LANGUAGES` 数组中注册**：`['zh', 'en', 'ja']`
2. **在 `detectLang()` 中添加检测分支**：`return nav.startsWith('ja') ? 'ja' : ...`
3. **在 `T` 对象中新增语言层**：复制 `en` 层所有键，将值翻译为日语
4. **在 `setLang()` 中添加 `lang` 属性映射**：`document.documentElement.lang = lang === 'ja' ? 'ja' : ...`

最耗时的是第 3 步——需要翻译约 80 个键值对和若干函数返回值。**`t()` 和 `tLang()` 的回退逻辑（缺失键走中文）保证了即使翻译不全，页面也不会崩**，只会显示部分中文。
