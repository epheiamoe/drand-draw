# 安全边界与参数限制

任何系统的安全设计都不能仅靠代码防御，还需要明确的**边界契约**——告诉使用者："参数在这个范围内，算法保证正确；超出这个范围，结果不可预期。"drand-draw 的参数上限写死在算法规范中，既是工程约束，也是安全承诺。

---

## 边界总览

算法规范中定义了四条硬性上限：

| 参数 | 最大值 | 设计依据 |
|------|--------|----------|
| **N**（参与人数） | 1,048,575 | 20 位二进制上限（2²⁰ − 1） |
| **奖项层级** | 15 层 | 短码编码效率 + UI 空间 |
| **每层人数** | 1,000 | 短码长度 + 总奖数不得超过 N |
| **deadline** | 2106 年 | 32 位 Unix 时间戳上限 |

[来源](ALGORITHM.md#L155-L159)

---

## 逐条解释

### N ≤ 1,048,575：20 位二进制的选择

这个数字不是随意取的。1,048,575 = 2²⁰ − 1，恰好是 **20 位二进制的最大值**。选这个值的理由有三层。

**第一，短码紧凑性。** N 在短码中用 **base36** 编码。1,048,575 的 base36 表示为 `mh33`（4 个字符），即便在最坏情况下也只占 5 个字符。如果 N 达到 2²⁴（约 1,600 万），base36 编码就会扩展到 5-6 个字符，短码整体长度随之膨胀，不利于社交平台分享。[来源](src/encode.js#L9-L12)

**第二，BigInt 与 Number 的转换边界。** 中奖计算中有一个关键操作：

```javascript
const bigVal = BigInt('0x' + seedHex)      // SHA-256 → 256位 BigInt
let idx = Number(bigVal % BigInt(n))        // 取模后转换回 Number
```

`bigVal % BigInt(n)` 的结果是一个小于 n 的 BigInt，随后通过 `Number()` 转换回 JavaScript Number。**Number 的安全整数范围是 2⁵³，但取模结果小于 n。** 当 n ≤ 2²⁰ 时，这个值远小于 2⁵³，转换绝对安全。[来源](src/lottery.js#L30-L31)

**第三，实际场景足够。** 社交媒体抽奖的参与者通常在几千到几十万之间。100 万的上限覆盖了绝大多数真实活动，再大就超出了"公平抽奖"的社会信任边界——连如何验证参与者真实性都成了问题。

### 奖项层级 ≤ 15：UI 与算法的平衡

15 层奖项（一等奖到十五等奖）的设计纯粹是**实用主义决定**。

在前端 `CreateDraw` 组件中，每一层奖项对应一个动态生成的输入行：

```javascript
container.querySelector('#add-prize-btn').addEventListener('click', () => {
  const rows = prizesContainer.querySelectorAll('.prize-row')
  const idx = rows.length       // 当前层数
  // ... 生成新行
})
```

代码中没有硬编码 15 的检查——上限来自算法规范中的约定，而非 UI 拦截。理论上用户可以添加 16 层、20 层，但算法规范将此列为安全边界：超过 15 层的抽奖不在算法一致性承诺范围内。[来源](src/components/CreateDraw.js#L98-L108)

从短码角度看，每增加一层奖项，短码中就多一个 base36 编码的数字和分隔符。15 层已足够覆盖"一等奖到入围奖"的典型结构。

### 每层 ≤ 1,000：防膨胀

前端 HTML 中直接限制了输入范围：

```html
<input type="number" min="1" max="1000" value="1"
       class="prize-count ..." />
```

每个奖项最多 1,000 人的限制与 **总奖数不得超过 N** 的算法约束协同工作。如果某人设了 15 层 × 1,000 人 = 15,000 个奖项，但 N 只有 100, ，`computeWinners` 会直接抛错：

```javascript
if (totalPrizes > n) {
  throw new Error(`Total prizes (${totalPrizes}) exceeds N (${n})`)
}
```

[来源](src/lottery.js#L19-L21)

Python CLI 实现适用同样的逻辑。[来源](cli/drand_draw/lottery.py#L16-L18)

### deadline ≤ 2106：32 位时间戳的遗产

deadline 在短码中编码为**小写十六进制**：[来源](src/encode.js#L8)

```javascript
parts.push(params.deadline.toString(16))
```

32 位无符号整数的最大值是 **4,294,967,295**（十六进制 `ffffffff`，恰好 8 个字符），对应格林尼治时间 2106-02-07 06:28:15。如果 deadline 超出这个范围，十六进制字符串会变成 9 位或更长，短码稳定性被破坏。

为什么是 32 位？因为 drand 网络的 round 计算基于 Unix 时间戳，drand API 本身也使用 32 位时间戳体系。虽然 JavaScript 的 `Date` 和 Python 的 `int` 都支持更大的范围，但短码编码格式保持与 drand 基础设施一致。

> 实际场景中也不会有人设置 2106 年以后的抽奖截止日——那已超出了任何合理活动的生命期。

## BigInt 的跨平台处理差异

JavaScript 和 Python 在处理大整数时有一条微妙的裂缝。

**Python** 的 `int` 是原生任意精度，所有操作在同一类型中完成：

```python
seed_hex = hashlib.sha256(data).hexdigest()  # 64字符十六进制字符串
big_val = int(seed_hex, 16)                   # 任意精度整数
idx = big_val % n                             # 取模，结果仍然是 int
```

[来源](cli/drand_draw/lottery.py#L14-L21)

**JavaScript** 的情况不同。SHA-256 输出的 256 位数超过了 `Number.MAX_SAFE_INTEGER`（2⁵³ − 1），必须使用 `BigInt`：

```javascript
const seedHex = await deriveSeed(randomness, shift)  // 64字符十六进制
const bigVal = BigInt('0x' + seedHex)                 // BigInt
let idx = Number(bigVal % BigInt(n))                  // 取模后转回 Number
```

关键差异在于最后一步：JS 需要**显式地**将 BigInt 取模结果转换回 Number，因为 `winners` 数组存储的是 Number（供后续 `used.has(idx)` 查找和数组索引使用）。如果 n 过大，`bigVal % BigInt(n)` 的结果超出 Number 安全范围，转换会丢失精度。

**这就是 1,048,575 作为上限的深层原因**——它确保 BigInt → Number 的转换绝对安全。Python 没有这个转换步骤，因此理论上可以支持更大的 n，**但为了跨平台算法一致性**，CLI 工具也遵循相同的安全边界。[来源](ALGORITHM.md#L155-L159)

## 边界之外的信任

这些参数上限共同构成一个安全声明："只要你的抽奖活动在这个范围内，结果可验证、可重现。" 超出边界的参数不会被算法明确拒绝（除了总奖数 > N 会抛错），但**失去了一致性保证**——你的抽奖结果可能在 JS 和 Python 实现之间不一致。

这是可验证抽奖的本质：不仅算法要对，参数的边界也要对。边界是契约的一部分。
