# 智能解析引擎

验证抽奖系统的用户输入接口面临一个独特挑战：同一组抽奖参数可能以短码、URL、社交媒体分享文本、手动输入键值对等截然不同的格式出现。`smartParse` 函数的设计目标就一句话——**不管用户粘贴什么，把抽奖参数提取出来**。它不要求用户遵循严格的输入格式，而是在出错前穷尽所有合理的解析路径。

## 设计意图

在引入 smartParse 之前，前端手动验证页面需要三段各自独立的解析逻辑：检测 `=`/`?` 走 `hashToParams`，检测短码走 `decodeShortCode`，兜底走 `hashToParams('#/?' + val)`。这种模式有三层问题：第一，调用方必须预先判断输入类型；第二，每种新格式都需要在多个入口加条件判断；第三，Python CLI 和前端 JS 无法共享同一个解析心智模型。

smartParse 用一个函数封装了 **五层递进解析**，调用方只需问一句"能解析吗"，无需关心内部策略。这一设计在 `src/main.js` 的验证事件中彻底简化了调用路径——从之前的 13 行条件分支缩减到 2 行：

```js
const params = smartParse(val)
if (params) { doVerify(params); return }
```

[来源](src/encode.js#L73-L99) | [来源](src/main.js#L162-L163)

Python CLI 的 `drand-draw parse` 命令同样使用同一套递进逻辑：

```python
result = encode.smart_parse(text)
```

[来源](cli/drand_draw/__main__.py#L122-L122)

## 五层递进解析策略

解析器按严格优先级依次尝试五种策略，一旦某层返回有效结果就短路返回。这种**贪婪截断**的策略保证了确定性：相同的输入始终得到相同的解析结果。

### 决策流程图

```mermaid
flowchart TD
    A[输入 val] --> B{val 为空?}
    B -->|是| Z[返回 null]
    B -->|否| C{符合短码前缀<br>/^[qde]-[0-9a-f]+-/ ?}
    C -->|是| D[decodeShortCode]
    D -->|成功| Z1[返回 params]
    D -->|失败| E
    C -->|否| E{正则匹配<br>短码模式?}
    E -->|是| F[decodeShortCode]
    F -->|成功| Z2[返回 params]
    F -->|失败| G
    E -->|否| G{包含 http(s) URL?}
    G -->|是| H[提取 URL<br>解析 hash/search]
    H -->|成功| Z3[返回 params]
    H -->|失败| I
    G -->|否| I{包含 # 或 ? ?}
    I -->|是| J[截取片段<br>hashToParams]
    J -->|成功| Z4[返回 params]
    J -->|失败| K
    I -->|否| K{包含 = ?}
    K -->|是| L[整段作为 key=value<br>hashToParams]
    L -->|成功| Z5[返回 params]
    L -->|失败| Z
    K -->|否| Z
```

### 第一层：直接短码（Direct Short Code）

验证输入是否以 `q-`、`d-` 或 `e-` 开头且第二段是十六进制数。如果匹配，直接调用 `decodeShortCode`。这是**最高优先级**，因为这是系统自产自销的格式，它的匹配成本最低、准确率最高。

```js
if (/^[qde]-[0-9a-f]+-/.test(val)) {
  const d = decodeShortCode(val)
  if (d) return d
}
```

这层使用的头部匹配 `/^[qde]-[0-9a-f]+-/` 故意宽松——它只检查前两段的格式，不校验第三段及之后的可选段。如果这层失败（比如用户输入的短码有损），不会立刻放弃，而是让第二层兜底尝试从字符串中提取完整短码。

[来源](src/encode.js#L75-L78)

### 第二层：文本中提取短码

在字符串中搜索短码模式的子串。这层专为社交媒体场景设计——当用户从推文、简介或其他包含上下文文本的内容中复制粘贴时，解析器仍然能提取出短码。

**正则表达式**：`/\b[qde]-[0-9a-f]+-[0-9a-z]+(?:-[0-9a-z,]+)*\b/`

分节解析：

| 片段 | 语义 |
|---|---|
| `\b` | 词边界，防止 `xq-1a2b-3c` 这种部分匹配 |
| `[qde]` | 链前缀：`q`(QuickNet)、`d`(Default)、`e`(EVMNet) |
| `-` | 字面连字符 |
| `[0-9a-f]+` | deadline 十六进制表示 |
| `-` | 字面连字符 |
| `[0-9a-z]+` | N 的 base36 表示 |
| `(?:-[0-9a-z,]+)*` | 零或多个可选段：prizes 列表（逗号分隔的 base36）和 winners 列表 |

**关于字符集**：第二段 deadline 使用 `[0-9a-f]`（仅十六进制），第三段 N 使用 `[0-9a-z]`（base36，含 g-z）。这种非对称字符集是短码格式的自然约束——deadline 是时间戳的十六进制，N 是参与人数的 base36。因此匹配器优先用 `[0-9a-f]` 约束第二段以减少误匹配。

[来源](src/encode.js#L80-L84)

### 第三层：URL 提取与解析

通过正则 `/https?:\/\/[^\s]+/` 从文本中提取 URL，然后用 `new URL()` 解析其 hash fragment 和 search query。这层的关键在于它**不依赖整段文本格式**——用户可能粘贴完整 URL、链接加文案、甚至部分 URL。

处理流程：
1. 提取 URL 字符串
2. 优先解析 hash fragment（`#/?chain=...&deadline=...&n=...`）
3. 若 fragment 为空则回退到 query string（`?chain=...`）
4. 要求 `chain`、`deadline`、`n` 三个字段**同时存在**才视为有效

[来源](src/encode.js#L86-L93)

### 第四层：Hash/Fragment 解析

如果前三层都失败，检查字符串中最后一个 `#` 或第一个 `?` 的位置，截取从该位置开始的子串交给 `hashToParams` 解析。这里使用 `Math.max(val.lastIndexOf('#'), val.indexOf('?'))` 而非简单的 `val.indexOf('#')`，因为：
- 最后一个 `#` 之后的 hash 才是浏览器实际使用的 fragment
- 如果文本中既无 `#` 也无 `?`，`qIdx` 为 `-1`，跳过本层

[来源](src/encode.js#L95-L98)

### 第五层：原始 key=value

最后一搏：检查输入是否含 `=`，如果有，直接将其作为 URL query string 格式送 `hashToParams` 解析。这层覆盖了用户手写输入 `chain=quicknet&deadline=12345&n=100` 这种最原始格式。

[来源](src/encode.js#L100-L103)

## 跨平台实现一致性

JS 版（`smartParse`）和 Python 版（`smart_parse`）在解析策略上完全对齐。核心差异仅在于语法层面：

| 层面 | JS（`src/encode.js`） | Python（`cli/drand_draw/encode.py`） |
|---|---|---|
| 函数组织 | 单一函数 + 内嵌注释 | 5 个 `_try_*` 私有函数 + 调度循环 |
| URL 解析 | `new URL()` + `URLSearchParams` | `urlparse` + `parse_qs` |
| 正则引擎 | JS native RegExp | `re.search` |
| 空值处理 | `if (!val) return null` | `if not val or not val.strip(): return None` |

Python 版将每层逻辑拆成独立的 `_try_shortcode`、`_find_shortcode_in_text`、`_extract_url`、`_try_fragment`、`_try_raw` 函数，通过 steps 列表驱动调度：

```python
steps = [_try_shortcode, _find_shortcode_in_text, _extract_url, _try_fragment, _try_raw]
for step in steps:
    result = step(val)
    if result is not None:
        return result
```

这种结构化拆分的优势是每层可以独立测试，也便于未来在调度链中间插入新的解析策略。

[来源](cli/drand_draw/encode.py#L196-L207)

| `e-1a2b3c-abc-5,3` | 含 prizes `[5, 3]` | 带可选段 |
| `q-1a2b3c-abc-5,3-1,2` | 含 prizes `[5, 3]` 和 winners `[1, 2]` | 完整格式 |

| `抽奖链接 q-17f3f0-2s 快来参与` | `quicknet, 1572848, 104` | 英文文案环绕 |

### 第三层：URL 解析

| 输入 | 预期结果 | 说明 |
|---|---|---|
| `https://drand.pages.dev/#/?chain=quicknet&deadline=1572848&n=104` | 正常解析 | 标准 SPA hash URL |
| `https://drand.pages.dev/?chain=quicknet&deadline=1572848&n=104` | 正常解析 | 无 hash 的 query string |

|---|---|---|
| `#/?chain=quicknet&deadline=1572848&n=104` | 正常解析 | 纯 hash fragment |
| `?chain=quicknet&deadline=1572848&n=104` | 正常解析 | 纯 query string |
| `一些文字?chain=quicknet&deadline=1572848&n=104` | 正常解析 | 文本后接 query |

### 第五层：原始 key=value

| 输入 | 预期结果 | 说明 |
|---|---|---|
| `chain=quicknet&deadline=1572848&n=104` | 正常解析 | 最原始键值对 |
| `chain=quicknet&deadline=1572848&n=104&prizes=5,3` | 含 prizes | 带可选字段 |
| `chain=quicknet&deadline=abc&n=104` | 失败 | deadline 非数字 |

### 边界情况

| 输入 | 预期结果 | 说明 |
|---|---|---|
| 空字符串 `""` | null | 空值处理 |
| `undefined` | null | falsy 值处理 |
| `仅文字无匹配` | null | 无任何可解析内容 |
| `q-` | `decodeShortCode` 返回 null | 短码不完整，各层均失败 |
| `chain_only&deadline=1572848&n=104` | 失败 | 缺少 `=`，第五层跳过 |
| `#` 或 `?` 单独出现无参数 | 失败 | fragment 解析后 `hashToParams` 返回 null |

## 与短码编解码的关系

smartParse 是**消费侧**的入口函数，它依赖底层 `decodeShortCode` 和 `hashToParams` 两个解码函数。`decodeShortCode` 负责将连字符分隔的短码字符串转换为结构化参数对象，`hashToParams` 负责将 query string 格式的 hash fragment 解码。关于短码的编码细节，参见 [短码编解码规范](短码编解码规范.md)；关于短码在社交平台分享中的防限流策略，参见 [短码分享机制](短码分享机制.md)。
