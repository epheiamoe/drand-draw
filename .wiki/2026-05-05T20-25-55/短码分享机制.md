## 为什么需要短码

Twitter、微博等社交平台对外部链接普遍有**限流机制**——包含完整 URL 的推文会获得更少的自然曝光。如果直接在活动推文中附上抽奖验证链接，链接的可发现性会被算法压制。

短码方案的核心思路是：将抽奖参数压缩为一串**无特殊字符的短标识符**，让博主可以将其混在正文中自由传播，粉丝拿到后无论在哪个渠道都能还原为完整的验证参数。

## 短码格式

短码由五段组成，按 `-` 分隔：

```
{chain}-{deadline_hex}-{N_base36}-{prizes_base36}-{winners_base36}
```

| 段位 | 含义 | 编码方式 | 可选？ |
| --- | --- | --- | --- |
| 第 1 段 | drand 链标识 | 单字母缩写：`q`（quicknet）、`d`（default）、`e`（evmnet） | 必选 |
| 第 2 段 | 截止时间戳 (Unix 秒) | 十六进制小写 | 必选 |
| 第 3 段 | 参与者总数 N | base36 | 必选 |
| 第 4 段 | 各奖项层级中奖人数 | 逗号分隔，每人次用 base36 | 可选（开奖后） |
| 第 5 段 | 中奖者序号 | 逗号分隔，每个序号用 base36 | 可选（开奖后） |

例如一条完整的短码：

```
q-64f5a30-3e8-1,2-a,2d,52
```

它展开后的含义是：

- **链**：quicknet（`q`）
- **截⽌**：`0x64f5a30` = 1692800000 (Unix 秒)
- **总人数**：`3e8` (base36) = 1000 人
- **奖项**：一等奖 1 人，二等奖 2 人
- **中奖者**：10 号 (`a`)、45 号 (`2d`)、82 号 (`52`)

## 对比：URL 参数格式 vs 短码格式

除了短码，系统同样支持 **URL 哈希参数格式**。两种格式服务于不同的场景。

| 维度 | URL 参数格式 | 短码格式 |
| --- | --- | --- |
| 示例 | `#/?chain=quicknet&deadline=1692800000&n=1000&prizes=1,2&winners=10,45,82` | `q-64f5a30-3e8-1,2-a,2d,52` |
| 可读性 | 字段名自解释，人类可读 | 需要解码才能理解 |
| 紧凑性 | 约 100+ 字符 | 约 25–40 字符 |
| 链接特性 | 完整的 URL 片段，浏览器可直接导航 | 纯文本片段，不构成链接 |
| 传播友好度 | 社交平台识别为外链，触发限流 | 无特殊字符，不被视为链接 |
| 编码方式 | 十进制数字、明文链名 | base36 + hex，链名缩写为单字母 |
| 解码方式 | `URLSearchParams` 解析 | `decodeShortCode()` / `decode_short_code()` |

前端的 `paramsToHash()` 函数生成 URL 参数格式：

```js
// 输出: #/?chain=quicknet&deadline=1692800000&n=1000&prizes=1,2&winners=10,45,82
export function paramsToHash(params) {
  const usp = new URLSearchParams()
  usp.set('chain', params.chain)
  usp.set('deadline', String(params.deadline))
  usp.set('n', String(params.n))
  // prizes、winners 以逗号分隔的十进制字符串追加
  return '#/?' + usp.toString()
}
```

`encodeShortCode()` 生成短码。两套格式可以相互无损转换，前端通过 `hashToParams()` 统一处理：入口先尝试以 URLSearchParams 解析，若失败则尝试短码解码。

## 编码设计理由

### 为什么用 base36 编码整数

N（参与者总数）最大可达 10,000,000（一千万），十进制表示需要 8 位字符。base36 使用 `0-9a-z` 共 36 个字符编码数字，比十六进制（16 个字符）更紧凑，又比 base64（含 `+`、`/` 等特殊符号）更安全——不会与短码的分隔符 `-` 冲突。

对比 `N = 10000000` 的三种编码长度：

| 进制 | 编码结果 | 字符数 |
| --- | --- | --- |
| 十进制 | `10000000` | 8 |
| 十六进制 | `989680` | 6 |
| **base36** | **`5yc1s`** | **5** |

同样的逻辑应用于奖品层级人数和中奖者序号——这些小整数经过 base36 编码后几乎总是单字符，短码整体能压缩 60% 以上。

[来源](cli/drand_draw/encode.py#L11-L21)

### 为什么用十六进制编码时间戳

截止时间戳是 Unix 秒数，典型值在 1.7–2.0×10⁹ 范围，十进制需要 10 位。十六进制只需 8 位，压缩比 20%。更重要的是，时间戳不像 N 会动辄达到百万级——base36 带来的额外增益极小，而十六进制的可读性更好，一眼能辨认出这是时间值而非序号。

```
十进制: 1692800000  (10 字符)
十六进制: 64f5a30  (7 字符)
```

### 为什么用单字母缩写表示链名

三条支持链——quicknet、default、evmnet——分别是 8、7、6 个字符。用 `q`、`d`、`e` 三个单字母替代，短码首段固定为 1 字符，为后续字段留出更多空间，也方便肉眼区分。映射表定义在 `chains.js` 和 `encode.py` 的 `CHAINS` 字典中：

```
quicknet → q    default → d    evmnet → e
```

## 编解码往返测试

两个平台各自实现了往返无损的编解码逻辑。以下展示四组典型用例：

### JavaScript (前端)

```js
// 编码
const params = {
  chain: 'quicknet',
  deadline: 1692800000,
  n: 1000,
  prizes: [1, 2],
  winners: [10, 45, 82]
}
const code = encodeShortCode(params)
// → "q-64f5a30-3e8-1,2-a,2d,52"

// 解码
const decoded = decodeShortCode(code)
// → {
//     chain: "quicknet",
//     deadline: 1692800000,
//     n: 1000,
//     prizes: [1, 2],
//     winners: [10, 45, 82]
//   }

// 无奖项/无中奖者（抽奖前）
encodeShortCode({ chain: 'default', deadline: 1730000000, n: 500 })
// → "d-6717ac0-eg"
```

[来源](src/encode.js#L3-L32)

### Python (CLI)

```python
from drand_draw.encode import encode_short_code, decode_short_code

code = encode_short_code('quicknet', 1692800000, 1000, [1, 2], [10, 45, 82])
# → "q-64f5a30-3e8-1,2-a,2d,52"

params = decode_short_code(code)
# → {
#     'chain': 'quicknet',
#     'deadline': 1692800000,
#     'n': 1000,
#     'prizes': [1, 2],
#     'winners': [10, 45, 82]
#   }
```

CLI 还提供了 `encode` 和 `decode` 两条子命令，可直接在终端操作：

```bash
# 编码
drand-draw encode --chain quicknet --deadline 1692800000 --n 1000 \
  --prizes 1,2 --winners 10,45,82
# 输出: q-64f5a30-3e8-1,2-a,2d,52

# 解码
drand-draw decode q-64f5a30-3e8-1,2-a,2d,52
# 输出:
#   Chain: quicknet
#   Deadline: 1692800000 (YYYY-MM-DD HH:MM:SS UTC)
#   N: 1000
#   Prizes: 1, 2
#   Winners: #10, #45, #82
```

### 往返不变性

无论用哪个平台编码，结果都能被另一平台解码，且解码后的参数与原始输入完全一致。这种跨平台兼容性依赖于两端共享的编码规范：

- 相同的链前缀映射表
- 相同的 base36 算法（Python 内置 `int(s, 36)` / JS `Number.prototype.toString(36)`）
- 相同的十六进制格式约定（小写、无 `0x` 前缀）

## 短码的传播路径

短码在系统中的完整生命周期是：

1. **博主创建抽奖** → 前端生成短码，显示在结果页面供复制
2. **博主分享** → 将短码嵌入推文正文，例如：
   > 开奖了！验证码 `q-64f5a30-3e8-1,2-a,2d,52`，去 drand-draw.pages.dev 查验
3. **粉丝验证** → 将短码粘贴到验证页面的输入框，或直接访问 `#/verify/{短码}`
4. **前端解析** → `smartParse()` 自动识别输入内容，无论输入的是短码、URL 还是纯文本，都能提取参数

相关机制的完整实现参见 [智能解析引擎](智能解析引擎.md)。
