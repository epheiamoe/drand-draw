# 项目路线图与开发计划

## 一句话定位

**drand-draw** 是一个基于 drand 去中心化随机信标网络的社交平台抽奖工具。博主设定截止时间、粉丝参与、截止后用 drand 随机数开奖、结果公开可验证。整个系统是纯静态前端页面，不运行任何后端服务器。 [来源](PLAN.md#L1-L4)

---

## 设计哲学：这不是区块链项目

这是理解项目的第一把钥匙。项目在多个地方反复强调：

> **这不是 Web3/NFT 项目，不是智能合约，不运行在区块链上，不存储任何数据。** [来源](README.md#L5-L7)

> **这不是区块链项目，不是 Web3 工具，不是 NFT mint，不是企业级随机数服务。** 这是一个给 X/Twitter 博主用的社交平台抽奖工具，仅此而已。 [来源](GUIDE.md#L3-L5)

为什么刻意划清界限？因为 **drand-draw 选择了更简单的技术路径**——核心随机数虽然来自去中心化网络，但整个抽奖逻辑（Round 计算、中奖编号推导、短码编解码）全部以**纯前端 JavaScript** 执行，不涉及 Gas 费、钱包签名、链上交易。用户只需要一个浏览器就能发起和验证抽奖，无需安装任何插件，也无需支付任何费用。 [来源](PLAN.md#L7-L8)

这种设计的好处是显而易见的：部署成本为零（Cloudflare Pages 免费额度）、用户门槛降到最低（有浏览器就行）、验证过程完全透明（任何人都可以打开链接看到完整计算过程）。 [来源](PLAN.md#L7-L8)

## 核心特性清单

项目 `PLAN.md` 定义了六个核心特性，均已实现：

| 特性 | 说明 | 实现状态 |
|------|------|----------|
| **完全透明** | 抽奖结果由 drand 公开随机数决定，任何人可独立验证 | ✅ 已实现 |
| **零后台** | 纯静态页面，部署在 Cloudflare Pages，不存储任何数据 | ✅ 已实现 |
| **中英双语** | 自动检测浏览器语言，手动可切换 | ✅ 已实现 |
| **多链支持** | 支持 quicknet(3s)、default(30s)、evmnet(3s) 三条 drand 网络 | ✅ 已实现 |
| **多奖项** | 支持一/二/三等奖等多层级配置，每层最多 1000 人 | ✅ 已实现 |
| **短码分享** | 支持 URL 和短码两种分享方式，避免外链限流 | ✅ 已实现 |

[来源](PLAN.md#L6-L13)

### 每条特性的具体含义

- **完全透明**：随机数来自 drand 网络，由 League of Entropy 的 15+ 独立机构共同产出，没有单一实体能操控。任何粉丝都可以打开同一链接，获得与博主完全一致的结果。 [来源](src/i18n.js#L73-L76)
- **零后台**：所有数据携带在 URL 参数或短码中，页面从 URL 解码即可恢复全部状态。没有数据库、没有 API 服务器、没有用户注册系统。 [来源](PLAN.md#L7-L8)
- **中英双语**：使用 `i18n.js` 中的键值对翻译系统，根据浏览器语言自动选择，用户也可通过界面按钮手动切换。 [来源](src/i18n.js#L1-L5)
- **多链支持**：三条 drand 链（QuickNet 3 秒间隔、Default 30 秒间隔、EVMNet 3 秒间隔）各有不同的创世时间和周期参数，在 `chains.js` 中集中配置。 [来源](src/chains.js#L1-L55)
- **多奖项**：算法支持最多 15 个奖项层级，每层最多 1000 个中奖名额，每层独立计算、碰撞顺延。 [来源](ALGORITHM.md#L64-L78)
- **短码分享**：短码将链标识、deadline、N、奖项、中奖编号编码为 `q-66364280-2s-1,3-2a,f,4e` 格式，避开社交平台对外链的限流策略。 [来源](PLAN.md#L61-L66)

## 已实现功能清单

项目从第一个 commit（`e55e973`）就具备完整核心功能，后续迭代在以下维度持续完善：

**核心抽奖引擎**（`src/lottery.js`）：
- Round 计算：根据 deadline 和链参数计算出唯一的 drand round 编号
- SHA-256 种子派生：每个奖位使用 `SHA256(randomness + ':' + shift)` 独立性计算
- 碰撞顺延处理：当多奖项出现重复编号时自动顺延至下一个可用编号
- 安全边界校验：总奖项数超过 N 时抛出异常 [来源](src/lottery.js#L1-L42)

**drand API 集成**（`src/api.js`）：
- 三条 drand 链的 HTTP API 封装
- 多 relay 故障切换（api.drand.sh → api2.drand.sh → drand.cloudflare.com）
- 超时重试机制 [来源](PLAN.md#L10)

**短码编解码**（`src/encode.js`）：
- URL 参数 ⇄ 短码的双向转换
- base36 编码、小端十六进制 deadline 编码
- 含 winners 和不合 winners 两种格式 [来源](PLAN.md#L53-L66)

**三态页面渲染**：
- 截止前：显示倒计时、预计 round、奖项配置
- 已截止未开奖：显示等待状态，博主可点击触发开奖
- 已开奖：展示 drand round、随机数、各奖项中奖编号、完整计算过程 [来源](PLAN.md#L43-L51)

**候选列表排序工具**（`src/components/SortTool.js`）：
- 支持 @handle、逗号、空格、换行多种分隔格式输入
- ASCII 升序排序 + 自动编号
- 导出为文本或图片（支持"一行一列"和"尽量正方形"两种模式） [来源](PLAN.md#L67-L72)

**CLI 命令行工具**（`cli/`）：
- Python 实现的独立命令行版本
- 五条命令：`verify`、`compute`、`encode`、`decode`、`parse`
- 核心算法与网页版保持**完全一致**（使用相同的测试向量验证） [来源](README.md#L48-L55)

## 攻击面分析（已防御的 vs 仍需社会约束的）

关于攻击面的完整讨论请参见 [攻击面分析与信任模型](攻击面分析与信任模型.md)，这里只做路线图层面的概述。

项目在设计时就已经系统性地考虑了六种攻击场景，并在代码层面做了对应防御：

| 攻击方式 | 防御机制 | 防御层级 |
|----------|----------|----------|
| 博主看到随机数后选有利的 round | Round 由 deadline 唯一确定，deadline 在 URL/短码中锁定 | 代码强制 |
| 博主对不同的验证者说不同的 N | N 在 URL/短码中锁定，所有人看到同一个数 | 代码强制 |
| 博主使用不同的排序规则 | 排序规则由页面硬编码声明（ASCII 升序） | 代码声明 |
| 博主篡改 drand 返回的随机数 | drand 结果由多节点签名，可用 drand-client 验证 | 网络层 |
| 博主多次开奖直到满意 | 同一个 URL/短码只能产出一个结果 | 算法确定 |
| 博主对同一抽奖多次验证说不同结果 | 页面自动验证，粉丝可自行操作 | 代码强制 |

[来源](PLAN.md#L17-L30)

**仍需社会约束的**（也是项目明确的已知限制）：

1. **博主必须在截止前公开 URL/短码** — 否则无法证明 deadline 没有被修改过
2. **博主必须在开奖前公开完整候选列表** — 粉丝需自行核对名单是否正确
3. **候选列表的去重由博主诚信处理** — 同一用户多次互动算一次，这无法通过代码强制执行

[来源](PLAN.md#L32-L36)

这意味着：**技术能解决的，代码已经解决了；技术不能解决的，需要博主和粉丝共同参与维护。**

## 未来可能的改进方向

> 以下内容基于项目现状和已知限制的推演，**不构成项目承诺**，仅作为贡献者参考。

### 1. 候选列表的链上承诺（缓解社会约束）

目前最大的社会约束是"博主必须在开奖前公开完整候选列表"。一种可能的改进是：博主在创建抽奖时对候选列表的哈希值做公开承诺（例如发推附带哈希），开奖后再公布完整列表供核对。这样即使博主事后篡改名单，哈希值会暴露不一致。

### 2. 多平台排序规则扩展

目前排序规则硬编码为 ASCII 升序（`数字 → 大写字母 → 小写字母`），这适用于 X/Twitter 的 @username 格式。如果未来扩展到其他平台（如豆瓣、B站），可能需要不同的排序策略。

### 3. 支持自定义奖项名称

目前奖项名称固定为"一等奖/二等奖"（或英文的 "1st Prize/2nd Prize"），不支持自定义名称如"特等奖""阳光普照奖"等。

### 4. 批量生成抽奖

对于需要频繁举办抽奖的博主（例如每周活动），可能希望有批量生成链接/短码的功能。

### 5. 更多 drand 链的支持

目前支持三条链，drand 网络未来可能增加新链（如更短周期的链），届时只需在 `chains.js` 中添加新配置即可。 [来源](src/chains.js#L1-L55)

### 6. 优化验证结果的可分享性

目前验证结果以文本形式呈现，未来可以生成更精美的验证卡片图片，方便粉丝截图分享。
