# 候选列表排序工具实现

抽奖结束后，博主需要将参与者列表以排序编号形式发布到社交平台，以证明过程透明、接受公众校验。**SortTool** 组件提供了一个完全在浏览器端运行的 Canvas 图像引擎：它接收原始参与者列表，自动排序去重，然后渲染为排版工整的 PNG 图片，支持按需分页和两种版式。

---

## 排版参数体系

`renderListImages` 函数内部定义了一组硬编码常量，构成整张输出画布的 **排版网格**：

| 常量 | 值 | 作用 |
|---|---|---|
| `PAD` | 24px | 内容区域左右内边距 |
| `W` | 600px | 画布固定宽度 |
| `ROW_H` | 30px | 单行高度 |
| `HEADER_H` | 48px | 顶部页眉区高度 |
| `FOOTER_H` | 32px | 底部页脚区高度 |
| `MAX_H` | 8000px | 单张画布最大允许高度（分页阈值） |
| `NUM_W` | 44px | 行号区域宽度 |
| `FONT_FAMILY` | `'SFMono-Regular, Consolas, ...'` | 等宽字体系列回退 |

这些常量并非随意选取：`W = 600px` 是 Instagram、X (Twitter) 等社交平台图片预览的最佳宽度，不会因压缩而模糊；`ROW_H = 30px` 配合 14px 字号留出 16px 行间距，确保长列表在手机上仍可阅读。`NUM_W = 44px` 经过测量可以容纳 6 位数序号（最大 999,999）带小数点而不溢出。

[来源](src/components/SortTool.js#L83-L89)

## 双模式渲染引擎

SortTool 提供两种排版模式，由调用者通过 `mode` 参数选择，对应 UI 上的「一行一列」和「尽量正方形」两个按钮。

### Column 模式（单列）

按固定顺序从上到下逐行排列，每行包含：**序号** + **@handle**。

```
┌──────────────────────────────────────┐
│  Participant List                     │  ← HEADER_H (48px)
│  Total: 100                1 / 2      │
├──────────────────────────────────────┤
│  0.    @alice                        │  ← ROW_H (30px) 每行，斑马色交替
│  1.    @bob                          │
│  2.    @charlie                      │
│  ...                                 │
│  49.   @zoe                          │
├──────────────────────────────────────┤
│         drand-draw.pages.dev         │  ← FOOTER_H (32px)
└──────────────────────────────────────┘
```

每页可容纳的行数由 `⌊(MAX_H - HEADER_H - FOOTER_H) / ROW_H⌋` 计算得出，`600px × (HEADER_H + n × ROW_H + FOOTER_H)` 的画布高度随行数动态调整，避免浪费像素。[来源](src/components/SortTool.js#L95-L96)

### Square 模式（网格）

将参与者排列成 **接近正方形的网格**，列数由 `Math.ceil(Math.sqrt(handles.length * CELL_H / CELL_W))` 计算。这个公式从宽高比出发：

```
单元格宽高比 = CELL_W / CELL_H ≈ 6.28
目标网格宽高 ≈ (cols × CELL_W) / (rows × CELL_H)
令 cols × rows = N → rows = N / cols
目标宽高比 → (cols² × CELL_W) / (N × CELL_H) ≈ 1
→ cols ≈ √(N × CELL_H / CELL_W)
```

每格 176px × 28px，字号降为 12px，行号用 10px，适配网格空间。[来源](src/components/SortTool.js#L147-L152)

## 自动分页逻辑

Canvas 并非无限画纸。浏览器对 Canvas 尺寸有 **隐含上限**：Edge/Chrome 在 width/height 超过 65535px 时抛出 `NS_ERROR_OUT_OF_MEMORY`，而 Safari 在高度超过 8,192px 时渲染异常。SortTool 采取了比浏览器限制更保守的 `MAX_H = 8000px`。

```
┌──── 计算逻辑 ──────────────────────────────┐
│ column 模式:                                │
│   itemsPerPage = ⌊(8000 - 48 - 32) / 30⌋   │
│                = ⌊7920 / 30⌋ = 264          │
│   pages = ⌈N / 264⌉                        │
│                                             │
│ square 模式:                                │
│   maxRowsPerImage = ⌈(2000 - 48 - 32) / 28⌉│
│                    = ⌈1920 / 28⌉ = 69       │
│   但 square 实际限制是 2000px，比 column 更  │
│   保守——因为网格场景下每屏呈现的信息量更少。 │
└─────────────────────────────────────────────┘
```

分页信息在页眉右上角显示为 `{p+1} / {pages}`，仅当 `pages > 1` 时渲染，避免单页面场景的视觉干扰。[来源](src/components/SortTool.js#L115-L119)

## 暗色主题配色方案

整个渲染引擎使用 **固定色值**（而非 CSS 变量），确保输出图片在不同设备、不同主题下一律呈现深色风格：

| 用途 | 色值 | 说明 |
|---|---|---|
| 背景底色 | `#030712` | 极深灰黑，接近纯黑但保留质感 |
| 行斑马色（偶数行） | `#0f172a` | 最暗的 slate 蓝 |
| 页眉渐变起始 | `#0f172a` → `#1e293b` | 线性渐变 |
| 标题文字 | `#60a5fa` | 蓝色 400，品牌色 |
| 序号文字 | `#4b5563` | 灰色 600，低层级信息 |
| 昵称文字 | `#e5e7eb` | 灰色 200，高对比主文字 |
| 分割线 | `#1f2937` | 仅 0.5px 宽，极细 |
| 页脚文字 | `#374151` | 中性灰，用于域名水印 |

这种配色策略源于一个关键设计决策：**输出图片需在任何背景（深色/浅色）下都可辨识**。使用暗色底 + 浅色字的通用组合，比适应用户当前主题更可靠。[来源](src/components/SortTool.js#L107-L113, L124-L138)

## 文字截断算法

当 handle 过长时，引擎通过 `CanvasRenderingContext2D.measureText()` 逐字符截断并追加省略号「…」：

```
function truncateText(ctx, text, maxWidth) {
  let displayText = text
  if (ctx.measureText(displayText).width > maxWidth) {
    while (ctx.measureText(displayText + '…').width > maxWidth
           && displayText.length > 1) {
      displayText = displayText.slice(0, -1)
    }
    displayText += '…'
  }
  return displayText
}
```

算法的核心特征是 **先测量完整文本宽度**，仅在超限时触发逐字符收缩，因此绝大多数短 handle 只付出一次 `measureText` 的代价。

Column 模式下 `maxWidth = W - 2×PAD - NUM_W - 12 = 600 - 48 - 44 - 12 = 496px`；Square 模式下 `maxWidth = CELL_W - 36 = 176 - 36 = 140px`。Square 模式因单元格狭窄，handle 截断几乎是必然的，但社交平台用户名通常不超过 15 字符（`@` + 14 字），在 12px 等宽字体下约 130px，恰好卡在阈值附近。[来源](src/components/SortTool.js#L131-L137)

## PNG 导出管线

引擎使用 Canvas 标准 API 完成导出，无任何第三方依赖：

```
async function canvasToBlobAsync(canvas) {
  return new Promise(resolve => canvas.toBlob(resolve, 'image/png'))
}
```

返回的 `Blob` 通过 `URL.createObjectURL` 转为 data URL，嵌入 `<img>` 标签预览，同时作为 `<a download>` 的 `href` 提供下载。文件名格式为 `participants-{page}.png`。

这里有一个值得注意的工程选择：`canvas.toBlob` 的回调式 API 被包裹为 Promise，便于 `generateImages` 函数用 `await` 等待所有页面渲染完成后一次性展示——避免逐个追加导致的 Layout Shift。[来源](src/components/SortTool.js#L74-L76)

## 渲染示例

以下分析展示不同规模参与者列表的渲染结果。

### 10 人 → Column 模式

```
600px × (48 + 10×30 + 32) = 600px × 380px
```

单页容纳，无需分页。画布高度约 380px，适合嵌入推文。

### 300 人 → Column 模式

```
每页 = 264 行 → 300 人需 2 页
Page 1: 600px × (48 + 264×30 + 32) = 600px × 8000px
Page 2: 600px × (48 + 36×30 + 32) = 600px × 1160px
```

第一页恰好达到 `MAX_H` 限制，第二页高度回归动态计算。

### 300 人 → Square 模式

```python
CELL_W = 176, CELL_H = 28
cols = ceil(sqrt(300 × 28 / 176)) ≈ 7
rows = ceil(300 / 7) = 43
maxRowsPerImage = 69  # 基于 2000px 限制
# 43 < 69 → 单页
imgW = 48 + 7 × 176 = 1280px
imgH = 48 + 43 × 28 + 32 = 1284px
```

网格模式在 `N ≤ 7 × 69 = 483` 时保持单页，列数随 N 增大从 2 到约 25 动态调整，防止单行过长。

### 1,000 人 → Square 模式

```
cols ≈ ceil(sqrt(1000 × 28 / 176)) ≈ 13
rows = ceil(1000 / 13) = 77
rowPages = ceil(77 / 69) = 2 页
```

两页纵向分页，每页显示一个完整的网格块，页眉标记页码。

## 与系统其他模块的关系

SortTool 的功能定位与系统架构有清晰的边界：

- **[系统架构全景](系统架构全景.md)**：SortTool 是纯前端工具，无任何 API 调用，符合「无后端服务」的架构原则。
- **[博主操作指南](博主操作指南.md)**：SortTool 在 `CreateDraw` 组件中以 `details/summary` 的可折叠面板形式承载，渲染入口为 `renderSortTool(sortContainer)`，作为创建抽奖页面的附属功能存在。[来源](src/components/CreateDraw.js#L195)
- **参与者解析逻辑**：`parseHandles` 函数内嵌在 SortTool 中，负责将 `@handle`、逗号、空格、换行分隔的原始文本清洗去重、按字母排序，与[智能解析引擎](智能解析引擎.md)的设计思路一脉相承，但聚焦于纯粹的列表解析。[来源](src/components/SortTool.js#L63-L71)

## 工程设计的权衡

回顾整个实现，有几个隐含的设计决策值得注意：

1. **固定宽度 vs 响应式宽度**：600px 的宽度是「一刀切」选择，牺牲了大屏设备上的横向空间利用率，换来了输出图片在不同平台间的一致性。
2. **客户端渲染 vs 服务端生成**：所有渲染在浏览器主线程完成，大列表（>5000 人）会短暂阻塞 UI，但省去了任何服务端成本和隐私担忧——参与者的 handle 列表从未离开用户设备。
3. **Canvas 原生 API vs 第三方库**：没有使用 `html2canvas`、`fabric.js` 等库，直接操作 Canvas 2D API。这意味着更高的性能、零依赖安装，但也意味着所有排版计算（文字换行、网格定位）都要手工完成。
