# 粉丝：验证抽奖结果

作为抽奖参与者，你不需要信任博主。只需要一个链接或短码，你就可以在几秒钟内独立验证抽奖结果的真伪——而且任何人都能得出完全一致的结论。

## 三种验证入口

页面提供三种方式进入验证，任选一种即可。

| 方式 | 操作 | 适用场景 |
|------|------|----------|
| **粘贴链接** | 复制博主分享的 URL，粘贴到输入框 | 最常见，博主在社交平台通常发链接 |
| **输入短码** | 复制博主分享的短码字符串 | 当链接被平台限流或截断时 |
| **手动输入参数** | 在下方面板逐一填写链、Round、N、中奖编号 | 当只有文字公告、没有可点链接时 |

三种方式的入口在页面底部的"手动输入"区域上方都有对应的输入框。输入后点击"验证"按钮，页面会自动接管后续流程。

## 验证流程：三步走

无论从哪个入口进入，验证的底层逻辑完全相同。整个过程的代码集中在 `performVerification` 函数中：

### 第一步：解析参数

页面从你提供的链接或短码中提取以下关键参数：

- **chain** — 使用哪条 drand 链（quicknet / default / evmnet）
- **deadline** 或 **round** — 截止时间戳，或直接指定 round 编号
- **n** — 参与者总人数
- **prizes** — 各奖项人数配置（如 1 个一等奖 + 3 个二等奖）
- **winners** — 博主声明的中奖编号列表

如果参数不完整（例如只贴了一个网址但没有中奖编号），页面会显示"等待博主开奖"状态，而不是立刻进入验证。

[来源](src/components/DrawStatus.js#L1-L19)

### 第二步：拉取 drand 随机数

页面根据 deadline 自动计算出对应的 **round 编号**：

```javascript
round = Math.floor((deadline - genesisTime) / period) + 1
```

然后从三个 drand relay 服务器之一拉取该 round 的 **randomness**（随机数）。如果第一个服务器失败，自动尝试下一个（最多重试 5 次，每次间隔 2 秒）。

```javascript
// 核心：fetchWithRetry 最多尝试 5 次，每次等 2 秒
const beacon = await fetchWithRetry(params.chain, round)
```

这意味着即便某个 drand 节点暂时不可达，验证也不会中断。

[来源](src/components/DrawStatus.js#L7-L16)

[来源](src/api.js#L13-L24)

### 第三步：计算 vs. 声明，比对结果

拿到 randomness 后，页面执行 `computeWinners` 算法：

```javascript
const computed = computeWinners(beacon.randomness, n, prizes)
```

该算法将 randomness 与每个奖位的序号拼接，通过取模运算为每个奖项生成独立的中奖编号。碰撞时自动顺延到下一个编号。

然后将计算结果与博主声明的 winners 逐一比对：

```javascript
const match = computed.length === claimed.length 
          && computed.every((v, i) => v === claimed[i])
```

如果数组完全一致，验证通过；只要有一个编号不同，验证不通过。

[来源](src/lottery.js#L12-L34)

[来源](src/components/DrawStatus.js#L113-L117)

## 绿色标记 vs. 红色标记

验证结果以非常醒目的视觉信号呈现，你不需要读代码也能看懂。

### ✅ 绿色盾牌 — 验证通过（verifySuccess）

```
🛡️ 验证通过
计算结果与声明一致，抽奖结果真实有效

randomness: 0x7a3b...
第1等奖: #42
第2等奖: #15, #78, #33
```

绿色表示 **计算结果与博主声明的中奖编号完全匹配**。任何人在同一时间打开同一链接都会看到绿色标记，因为 drand 的随机数是公开且唯一的。

### ❌ 红色叉号 — 验证不通过（verifyFail）

```
❌ 验证不通过
计算结果与声明不一致，请核对输入信息

randomness: 0x7a3b...
第1等奖（声明）: #7
第1等奖（计算）: #42
```

红色表示 **计算出来的中奖编号与博主声称的不一致**。但要注意：这不一定代表博主作弊。常见原因有：

- 博主公开的参与者名单与你手上的名单不同（N 不一致）
- 输入参数有误（如 deadine 或 chain 复制错了）
- 博主尚未发布最终版名单（还在 Phase 2 核对中）

验证失败时，页面会同时展示"声明值"和"计算值"两列，供你手动对比差异。

[来源](src/components/DrawStatus.js#L119-L160)

## 为什么博主无法作弊？

`PLAN.md` 中记录了完整的攻击面防御矩阵，以下是针对粉丝最关心的五种作弊场景的防护机制：

| 博主想做什么 | 为什么做不到 |
|---|---|
| 看到 randomness 后选一个有利的 round | Round 由 deadline 唯一确定，deadline 在链接/短码中锁定，不可更改 |
| 对不同的粉丝说不同的 N | N 在链接/短码中锁定，页面直接解码显示，所有人看到的 N 一致 |
| 使用不同的排序规则 | 排序规则由页面硬编码声明（ASCII 升序），博主无法否认 |
| 篡改 drand 返回的随机数 | drand 返回结果由 League of Entropy 多节点签名，任何人可通过 drand client 独立验证签名 |
| 多次开奖直到满意 | 同一个链接/短码只能产出一个结果，因为 randomness 由 round 唯一确定，无法重来 |

## 手动验证：即使没有链接也能查

如果你只看到一段文字公告，例如：

```
链: quicknet
Round: 1234567
N: 100
中奖: 42, 15, 78, 33
```

你可以在页面底部"手动输入"区域填写：

1. 选择链：quicknet
2. Round 编号：1234567
3. 参与人数 N：100
4. 中奖编号：42,15,78,33

点击验证，页面会拉取对应的 drand randomness 并执行同样的比对流程。这和使用链接/短码验证的底层代码完全一致。

[来源](src/main.js#L133-L153)

## 验证背后的信任基础

整个验证不依赖任何中心化服务器。你只需要两样东西：

1. **drand 的公开随机数** — 由 15 个以上独立机构（Cloudflare、Protocol Labs、以太坊基金会等）共同签名产生，任何人都无法单方面操控
2. **开源的前端代码** — 所有计算逻辑都在浏览器中运行，你可以审查每一行代码来确认没有隐藏

你甚至可以在命令行中用 Python CLI 独立验证同样的结果，确保和浏览器中看到的一致。详见 [CLI 命令参考](cli-命令参考.md) 和 [算法规范独立文档](算法规范独立文档.md)。
