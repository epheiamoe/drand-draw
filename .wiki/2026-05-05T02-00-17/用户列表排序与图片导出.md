# 用户列表排序与图片导出

## 设计意图

抽奖的第一步是确定参与者池。博主在发起抽奖时，需要向粉丝公开排序后的完整参与者名单，以此建立透明基础：中奖结果由 **drand 随机信标** 与列表中每个用户的排序位置共同决定，任何事后插入或删除参与者的行为都会导致结果不匹配。`SortTool` 组件完成两项核心任务：将杂乱的输入解析为确定的排序列表，并将该列表渲染为可下载的 PNG 图片，供博主在开奖前公示。

该组件作为 [博主：发起一次抽奖](博主-发起一次抽奖.md) 流程的一个可选但关键的子面板，嵌入在 `CreateDraw` 页面的 `details` 折叠区中，默认收起。[来源](src/components/SortTool.js#L4-L10)

## `parseHandles`：解析与确定性排序

函数签名：

```js
function parseHandles(raw) { /* ... */ }
```

[来源](src/components/SortTool.js#L102-L110)

### 处理管线

输入字符串依次经过四个变换步骤，每一步都是**幂等**的（多次执行结果相同）：

| 步骤 | 操作 | 代码 | 效果 |
|------|------|------|------|
| 1. 分隔符归一化 | `replace(/[,，、\s]+/g, ' ')` | 将中文逗号、英文逗号、顿号、各类空白字符统一替换为空格 | 消除多源粘贴格式差异 |
| 2. 移除 @ 符号 | `replace(/@/g, '')` | 删除所有 `@` 字符 | 统一为裸 handle 存储 |
| 3. 切分与过滤 | `split(/\s+/).filter(h => h.length > 0)` | 按空白切分，跳过空串 | 确保每个元素非空 |
| 4. 去重 | `[...new Set(...)]` | 利用 `Set` 的唯一性约束 | 防止同一用户重复计数 |
| 5. ASCII 升序排序 | `sort((a, b) => a.localeCompare(b, 'en', { sensitivity: 'base' }))` | 英语 locale，`sensitivity: 'base'` 忽略大小写和变音符号 | 确定的全序排列 |

[来源](src/components/SortTool.js#L102-L110)

### `sensitivity: 'base'` 的语义

`localeCompare` 的 `base` 级别仅区分基本字符（a ≠ b），不区分大小写（A = a），不区分变音符号（é = e）。这确保了 `@Alice` 和 `@alice` 被视为同一用户（去重阶段已合并），且在排序时相邻排列，与平台（如 X/Twitter）的 handle 规则一致——handle 不区分大小写。

### 确定性保证

整个流程无随机因素：相同的输入字符串必定产生相同的输出数组。这是抽奖验证的基石——博主和粉丝可以独立调用 `parseHandles` 从同一份原始列表得出完全一致的有序数组。

## `renderListImages`：Canvas 图片生成

函数签名：

```js
async function renderListImages(handles, mode)
```

接受排序后的 handle 数组（不含 `@` 前缀）和模式名，返回 `Blob[]`，每个元素是一张 PNG 图片的二进制数据。[来源](src/components/SortTool.js#L114-L235)

### 全局常量

| 常量 | 值 | 含义 |
|------|-----|------|
| `PAD` | 24 | 左右内边距 |
| `W` | 600 | 图片宽度（像素） |
| `ROW_H` | 30 | 每行高度（column 模式） |
| `HEADER_H` | 48 | 表头区域高度 |
| `FOOTER_H` | 32 | 页脚区域高度 |
| `MAX_H` | 8000 | **单张图片最大高度**（浏览器 Canvas 硬限制） |
| `NUM_W` | 44 | 序号列宽度 |
| `FONT_FAMILY` | `'SFMono-Regular, Consolas, "Liberation Mono", Menlo, monospace'` | 等宽字体栈 |

[来源](src/components/SortTool.js#L115-L123)

### 单列模式（`mode === 'column'`）

**算法概要**：每页以 `HEADER_H` 高度开始，随后按 `ROW_H` 行高逐行填充，最后以 `FOOTER_H` 收尾。若行数超过单页容量，自动分页。

**单页容量计算**：

```
itemsPerPage = Math.floor((MAX_H - HEADER_H - FOOTER_H) / ROW_H)
             = Math.floor((8000 - 48 - 32) / 30)
             = Math.floor(7920 / 30)
             = 264 行/页
```

[来源](src/components/SortTool.js#L127)

**实际图片高度**：动态计算 `h = HEADER_H + pageItems.length * ROW_H + FOOTER_H`，避免留白。例如 100 个参与者共需 3 页（264 × 3 = 792，仅使用第一页的 100 行，高度为 48 + 100×30 + 32 = 3080px）。

**行渲染细节**：

- **交替行背景**：偶数行（`i % 2 === 0`）填充 `#0f172a`（深蓝灰），奇数行透明。
- **序号**：左对齐，`#4b5563`（灰），`12px` 字体，位置 `PAD + 4`。
- **Handle**：前置 `@`，`#e5e7eb`（浅灰），`14px` 字体，缩进 `PAD + NUM_W + 4`。
- **文字截断算法**：若 `measureText` 宽度超限，循环截断末尾字符直至添加 `…` 后满足宽度上限，保证不溢出。
- **分割线**：每行底部 `0.5px` 细线，`#1f2937`。

[来源](src/components/SortTool.js#L148-L176)

**表头**：渐变背景（`#0f172a` → `#1e293b`），蓝色标题 "Participant List"，灰色副标题 "Total: N"。多页时右上角显示页码。[来源](src/components/SortTool.js#L131-L147)

**页脚**：深色条居中显示 "drand-draw.pages.dev" 品牌标识。[来源](src/components/SortTool.js#L178-L183)

### 方形网格模式（`mode === 'square'`）

当参与者数量较大时，单列模式会生成很高很长的图片，不便于浏览。方形网格模式按**接近正方形的排列**排布。

**列数计算**：

```js
const cols = Math.max(2, Math.ceil(Math.sqrt(handles.length * CELL_H / CELL_W)))
```

其中 `CELL_W = 176`，`CELL_H = 28`。此公式的逻辑是：假设每格宽高比固定，求能使总面积接近正方形的列数。例如 1000 人：`sqrt(1000 × 28 / 176) ≈ sqrt(159.09) ≈ 12.6` → 13 列。

[来源](src/components/SortTool.js#L190)

**行分页**（区别于单列模式的行分页）：

网格模式下，单张图片高度受 `2000px` 限制（而非 `MAX_H`），这是硬编码值：

```js
const maxRowsPerImage = Math.ceil((2000 - HEADER_H - FOOTER_H) / CELL_H)
```

每张图片限制约 `2000px` 高度，是因为网格模式下行列数多，`8000px` 会产生过宽的图片，而 `2000px` 是社交媒体可滚动浏览的合理上限。[来源](src/components/SortTool.js#L192)

**格子渲染**：每个格子宽度 `CELL_W = 176`，高度 `CELL_H = 28`，序号与 handle 的排版逻辑与单列模式相同，只是字体缩小至 `10px`（序号）和 `12px`（handle），截断宽度阈值缩小为 `CELL_W - 36`。[来源](src/components/SortTool.js#L207-L230)

## 图片分页逻辑汇总

两种模式的分页策略不同，下表对比：

| 维度 | 单列模式 | 方形网格模式 |
|------|----------|--------------|
| 单页最大高度 | `MAX_H = 8000px` | `2000px`（硬编码） |
| 分页粒度 | 按行数 `itemsPerPage = 264` | 按行组 `maxRowsPerImage` |
| 图片宽度 | 固定 `600px` | 动态 `24×2 + cols×176` |
| 页面计数 | 右上角显示 `p+1/pages` | 右上角显示 `rp+1/rowPages` |
| 适用场景 | 人数少（<500），便于下载单张完整图 | 人数多，缩小单图高度便于社交平台展示 |

分页产生的每个 Blob 独立渲染、独立下载，文件名为 `participants-{page}.png`。[来源](src/components/SortTool.js#L126-L128)

## 组件集成与使用流程

`SortTool` 在 `CreateDraw` 页面渲染完成后被挂载到表单底部：

```js
const sortContainer = document.createElement('div')
container.querySelector('.max-xl').appendChild(sortContainer)
renderSortTool(sortContainer)
```

[来源](src/components/CreateDraw.js#L197-L199)

用户交互流程：

1. 在 `<details>` 折叠面板中粘贴参与者原始文本（支持 `@` 前缀、逗号、空格、换行混合分隔）。
2. 输入时实时调用 `parseHandles` 并更新预览区（默认带行号）。
3. 选择导出模式（单列 / 网格），触发 `generateImages` 异步函数。
4. `generateImages` 内部调用 `renderListImages`，通过 `canvas.toBlob` 将 Canvas 转换为 PNG Blob。
5. 每个 Blob 生成 `objectURL` 插入 DOM，附带下载链接和内嵌图片预览。[来源](src/components/SortTool.js#L115-L135)
6. "复制排序文本"按钮可将带行号的纯文本写入剪贴板。

## 在设计抽奖流程中的角色

博主在发起抽奖前，必须先**公开排序后的完整参与者列表**，这是整个安全模型的前提。[抽奖的安全模型](抽奖的安全模型.md) 中分析了此步骤的必要性：如果名单在抽奖后仍可变，博主可以在知道中奖索引后插入自己的账号。

具体约束如下：

- **开奖前**：博主将 `parseHandles` 输出的排序列表（通过图片或文本）公开。
- **开奖时**：`computeWinners` 使用该列表的**当前顺序**作为输入。[核心抽奖算法详解](核心抽奖算法详解.md) 中说明了 `computeWinners` 如何根据 drand 随机数和有序列表计算中奖者。
- **验证时**：粉丝用同一份原始列表调用 `parseHandles`，得到相同的排序结果，再配合 drand 信标验证中奖者是否匹配。

因此，`SortTool` 提供的不是可有可无的辅助功能，而是整个透明抽奖机制的**数据层基础设施**——它将非结构化、多格式的用户输入转化为严格的、可重现的全序排列。
