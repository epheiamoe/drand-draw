# 安全边界与参数限制

drand-draw 系统对每个核心参数都设定了硬性或软性上限。这些边界并非随意选取——每一条都对应着底层技术栈中某个不可逾越的约束：JavaScript 的数字精度、base36 编码的紧凑性、浏览器 Canvas 的渲染极限，或仅仅是工程上的务实判断。理解这些边界的成因，是在生产环境中正确使用本系统的前提。

## 参数上限总览

| 参数 | 最大值 | 技术约束链 |
|------|--------|-----------|
| N（参与人数） | 1,048,575（≈ 100 万） | BigInt 取模 → `Number()` 转换 → 短码紧凑性 |
| 奖项层级 | 15 层 | 实用性边界 + 短码段数限制 |
| 每层人数 | 1,000 | HTML input `max` 属性 + 实用考量 |
| deadline（截止时间） | 2106-02-07（Unix 无符号 32 位上限） | JavaScript `Date` 兼容性 + 16 进制编码长度 |
| Canvas 单图高度 | 8,000 px | 浏览器 Canvas 实现限制 |

## N ≤ 1,048,575：参与人数上限

### 这个数字从何而来

1,048,575 = 2²⁰ − 1。选择 20 位而非 32 位或 53 位，是工程权衡的结果。

### 三重约束

**第一层：BigInt 取模后的 Number 转换。** 核心算法 `computeWinners` 的关键行：

```javascript
let idx = Number(bigVal % BigInt(n))
```

`BigInt(n)` 本身可以处理任意大的 N，但取模结果必须通过 `Number()` 转回普通数字才能存入 `winners` 数组和 `used` Set。`Number` 安全整数范围是 [−2⁵³+1, 2⁵³−1]，即约 9 千万亿。理论上 N 可达 2⁵³，但 20 位远低于此限——这里并非 JavaScript 能力上限，而是有意留出的安全余量。

**第二层：短码编码的紧凑性。** N 在短码中以 base36 编码存储：`n.toString(36)`。1,048,575 的 base36 编码是 `1zt3f`（5 字符）。若将上限推至 1,000 万，base36 编码变成 7 字符；推至 1 亿，变成 8 字符。短码一旦过长会降低手动抄写的可用性，X/Twitter 等平台的链接预览也可能截断。

**第三层：碰撞解析的算法复杂度。** `computeWinners` 的碰撞处理是 O(总奖位数 × 碰撞率) 的线性扫描。当 N 较小时（相对于总奖位数），碰撞率上升，环形查找的迭代次数增加。N ≥ 2²⁰ 时，即使奖项总数高达 15,000（15 层 × 1000 人），碰撞概率仍在可控范围。若 N 缩小到数百而奖项总数也接近 N，极端情况下 `attempts >= n` 条件被触发，直接抛出 `Collision resolution failed` 错误。

[来源](src/lottery.js#L22-L25) · [来源](src/lottery.js#L29-L33) · [来源](src/encode.js#L5-L7)

### 超出上限的行为

URL 方式（`hashToParams`）：用 `parseInt(params.n)` 解析 N，无硬性上限检查——传入 2,000,000 也能解析，但页面后续渲染可能异常。短码方式（`decodeShortCode`）：同样用 `parseInt(parts[2], 36)` 解析 base36，无上限过滤。HTML 表单输入：`<input max="1000000">` 提供浏览器层的前端校验，但此约束不保护直接 URL 访问。

实际风险不在于系统崩溃，而在于短码过长失去可读性，以及极端碰撞场景下算法失败。

## 奖项层级 ≤ 15，每层 ≤ 1,000

### 15 层的实用边界

15 层不是一个技术硬上限，而是工程上的"足够"边界。从动机上说，社交平台抽奖场景中同时设置超过 15 个奖项层级（一等奖到十五等奖）的用户需求接近于零。代码中唯一与之相关的隐含约束是短码中 prizes 段落的长度——15 个逗号分隔的 base36 数字约 30-45 字符，仍在合理范围内。

中文本地化中的 `prizeTier` 函数：
```javascript
'i18n 映射': '第${'一二三四五六七八九十'[i]||(i+1)}等奖'
```
仅硬编码了 10 个汉字映射，超过 10 的层级退化为数字显示，但不限制功能。

[来源](src/components/CreateDraw.js#L93-L95)

### 每层 1,000 人的 HTML 级约束

表单中的每个奖项输入框都有 `max="1000"`：

```html
<input type="number" min="1" max="1000" value="1" class="prize-count ...">
```

这是一个 **客户端** 约束。直接通过 URL 或短码传入每层 10,000 人时，`computeWinners` 不会拒绝——它的唯一校验是 `totalPrizes > n`。这意味着：

- **通过表单**：浏览器阻止输入超过 1,000
- **通过 URL/短码**：可绕过，但会受 N 上限的间接约束（总奖位数不能超过 N）

[来源](src/components/CreateDraw.js#L47)

### 超出上限的行为

总奖位数超过 N 时，`computeWinners` 抛出 `Total prizes ({total}) exceeds N ({n})`；创建表单页会弹出 `alert` 提示。短码解码不会触发此错误——它在计算阶段才发生。

## deadline ≤ 2106 年：32 位无符号时间戳的上限

### 2106 年的由来

Unix 传统时间戳使用 **32 位有符号整数**，最大值 2³¹−1 = 2,147,483,647，对应 **2038-01-19 03:14:07 UTC**——即著名的 **Year 2038 问题**。若使用 **32 位无符号整数**，最大值 2³²−1 = 4,294,967,295，对应 **2106-02-07 06:28:15 UTC**。

drand-draw 的 deadline 编码采用 `deadline.toString(16)`，4,294,967,295 的十六进制为 `ffffffff`（8 字符）。这个值可以安全地通过 JavaScript `Number` 表示和 `parseInt(parts[1], 16)` 解析，没有任何运行时风险。

选择 2106 年而非 2038 年作为文档声明的上限，是因为 `Date(deadline * 1000)` 在 JavaScript 中能正确处理超出 32 位有符号范围的数字——无符号 32 位是 JavaScript `Date` 能全兼容的安全边界。

[来源](src/encode.js#L6) · [来源](src/encode.js#L22)

### 超出上限的行为

若 deadline 超过 4,294,967,295，`deadline.toString(16)` 产生的十六进制字符串超过 8 字符，短码段长度增加但不触发错误。`computeRound(chainId, deadlineUnix)` 中 `elapsed = deadline - chain.genesisTime` 仍能正常计算。唯一实际影响是倒计时显示——`new Date(deadlineMs)` 在浏览器中的行为是明确定义的，可处理百万年级别的时间戳。

### 跨链的 round 溢出问题

理论上，极高的 deadline 会导致 round 编号超出 drand 网络的实际出块范围——drand 网络本身可能早在 2106 年之前就停止运行或升级。此时 `fetchBeacon` 会因 HTTP 404 失败，多次重试后抛出 `All relays failed` 错误。

## Canvas 渲染的隐性边界

参与者列表图片导出功能（`SortTool`）涉及浏览器 Canvas 的两个硬限制：

| 参数 | 值 | 影响 |
|------|-----|------|
| `MAX_H`（最大高度） | 8,000 px | 单列模式每页最多 264 人 |
| Square 模式最大高度 | 2,000 px | 网格模式每页约 69 行 |
| 图片宽度 `W` | 600 px（固定） | 不随人数变化 |

8,000 px 是一个保守选择。大部分浏览器 Canvas 的实际高度上限是 32,767 px（Safari）或 65,535 px（Firefox/Chromium）。选择 8,000 是为了在所有主流浏览器上可靠工作，同时保持单张图片的文件大小可控。

当 N > 264（单列）时，系统自动分页生成多张 PNG 图片，每页底部显示页码和总数。

## 短码长度的间接约束

短码本身没有显式长度上限，但其各组成部分的编码特性决定了总长度范围：

| 字段 | 编码方式 | 最坏情况长度 |
|------|---------|-------------|
| 链前缀 | 字面字符 | 1 字符 |
| deadline | 小端十六进制 | 8 字符（`ffffffff`） |
| N | base36 | 5 字符（1,048,575 → `1zt3f`） |
| prizes | base36 + 逗号 | ~45 字符（15 层 × 3 字符 + 14 逗号） |
| winners | base36 + 逗号 | 随中奖人数线性增长 |

一个典型的短码（N=100 万，15 奖项，含中奖编号）大约 40–120 字符，远超社交平台的链接字符限制。短码的主要使用场景是手动抄写或二维码嵌入，120 字符在此场景下仍可接受。

[来源](src/encode.js#L3-L11)

## 扩展方向的限制因素

若要突破上述边界，需要相应的架构调整：

**支持更大的 N（如 1 亿）**：核心算法 `computeWinners` 中 `Number(bigVal % BigInt(n))` 不需要修改——`Number` 安全范围远大于 1 亿。真正的瓶颈在短码：N = 10⁸ 的 base36 是 `1njchs`（7 字符），仍然可接受。主要问题是碰撞解析的 O(奖位数) 循环——当奖项总数也增大时，`attempts >= n` 条件可能触发，需要将碰撞策略从线性探查改为更高效的方案（如 Fisher-Yates 洗牌 + 截取）。

**支持更多奖项层（超过 15）**：仅需移除前端 `<input max="1000">` 约束，算法本身无层数限制。短码中 prizes 段长度线性增长。

**突破 2106 年上限**：JavaScript 端无需修改；主要依赖 drand 网络在 22 世纪是否仍在运行——这是一个无法由系统本身保证的外部依赖。
