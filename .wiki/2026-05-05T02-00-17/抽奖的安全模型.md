# 抽奖的安全模型

一个去中心化抽奖系统的核心问题不是"代码能不能跑"，而是"博主能不能作弊"。drand-draw 的信任模型基于一个简单前提：**抽奖的每个关键参数都必须在开奖前锁定并公开，让粉丝能独立验证**。这个模型介于"完全信任博主"和"完全无需信任"之间，属于 **trustless with social constraints**——技术锁死博主篡改结果的能力，但博主仍有一些必须诚信处理的事情。

---

## 五类攻击方式与防御机制

以下表格列出了博主可能尝试的五种篡改手段，以及系统如何将其封死。

| # | 攻击方式 | 攻击描述 | 防御机制 | 防御类型 |
|---|---------|---------|---------|---------|
| 1 | **博主选择 favorableround** | 博主在 deadline 之后，先查看已公开的 drand randomness，再挑一个对自己有利的 round 声称是"应使用的 round" | **round 由 deadline 唯一确定**：`round = floor((deadline - genesis) / period) + 1`，deadline 在 URL/短码中锁定，不可更改 | 技术防御 |
| 2 | **博主对不同的粉丝说不同的 N** | 博主对粉丝 A 说"有 50 人参与"，对粉丝 B 说"有 100 人参与"，从而控制中奖编号落在自己想要的区间 | **N 在 URL/短码中锁定**：粉丝打开链接即能看到 N 的值，页面直接解码显示，博主无法抵赖 | 技术防御 |
| 3 | **博主使用不同的排序规则** | 博主声称"我是按字母序排的"，但实际用了另一种排序使特定用户编号不同 | **排序规则由页面硬编码声明**：ASCII 升序排列（数字 → 大写字母 → 小写字母），规则在博客公布中标明，粉丝可据此核对 | 技术防御 |
| 4 | **博主篡改 drand 返回的随机数** | 博主收到 drand 返回的随机数后，私自修改 randomness 值以操控中奖结果 | **drand 返回结果由 League of Entropy 多节点签名**，粉丝可通过 drand HTTP API 自行获取同一 round 的 randomness 与之比对 | 技术防御 |
| 5 | **博主多次开奖直到满意** | 博主在同一 round 上反复抽奖，直到出现想要的组合后才公布"最终结果" | **同一个 URL/短码只能产出一个结果**：round 由 deadline 固定，randomness 是固定的，computeWinners 是纯确定性算法，粉丝自行 computing 即可验证 | 技术防御 |

[来源](PLAN.md#L26-L37) | [ALGORITHM.md#L61-L87]

## 技术防御与社会约束的分界线

5 种攻击全被技术手段封死了，但信任模型并未达到 **fully trustless**。原因在于——技术只能保证"开奖过程不可篡改"，无法约束"开奖前的行为"。

### 需要社会约束的部分

| 约束项 | 为什么技术管不了 | 社会约束方式 |
|--------|----------------|-------------|
| **博主须在 deadline 前公开 URL/短码** | 技术无法阻止博主在 deadline 之后才公布链接——此时博主已知道 randomness，虽不能改 round，但可以选择不发布对自己不利的结果 | 博主应在发起抽奖时就公开链接，粉丝见不到提前公开链接的抽奖应保持怀疑 |
| **博主须公开完整候选列表** | 系统不知道有哪些用户参与了抽奖 | 博主须公布编号 0 ~ N-1 的完整用户名列表，粉丝自行核对自己的用户名是否在正确的位置 |
| **候选列表的去重** | 一个用户多次交互（如多次点赞）是否算多次，取决于平台规则 | 博主需诚信去重，粉丝可根据平台交互记录验证 |

[来源](PLAN.md#L39-L44)

用一句话概括：**技术保证"开奖那一刻之后，谁也改不了结果"；社会约束保证"开奖那一刻之前，博主没有藏着掖着"。**

## Trustless 还是 Fully Trustless？

drand-draw 是 **trustless in computation, but not fully trustless in process**。

- **Trustless in computation（计算无信任）**：任何人只要有同样的输入（deadline、N、prizeTiers），就能独立计算出完全一致的中奖编号。这一层不依赖任何人类诚信。
- **Not fully trustless（过程仍需信任）**：粉丝需要相信博主如实提供了候选列表、没有在 deadline 后才公布链接。这些属于"操作层面的信任"，代码解决不了。

对比区块链上的抽奖合约——链上抽奖把候选列表也 on-chain，自然能做到 fully trustless，但代价是复杂的合约开发和 Gas 费。drand-draw 选择了一个更轻量的折中方案：**用 drand 锁死随机数，用 URL 锁死参数，用社会约定约束操作流程。**

[理解-drrand-去中心化随机数](理解%20drand%20去中心化随机数.md) 解释了 drand 为什么能作为"不依赖任何单一方"的随机数源——League of Entropy 的多个独立节点运行分布式签名协议，只要至少一个节点诚实，随机数就不可预测。

## 安全边界：参数不是无限的

防御机制有效的前提是系统参数在安全范围内。ALGORITHM.md 定义了以下硬限制：

| 参数 | 最大值 | 设计考量 |
|------|--------|---------|
| N（参与人数） | 1,048,575 | 约 100 万上限，防止 BigInt 溢出和 URL 过长 |
| 奖项层级 | 15 层 | 从一等奖到十五等奖，再多对用户不友好 |
| 每层人数 | 1,000 | 防止奖品总数超出 N |
| deadline | 2106 年 | 对应 Unix 32 位时间戳上限，即 `0x7FFFFFFF` |

[来源](ALGORITHM.md#L100-L109)

超出这些边界时，`computeWinners` 会抛出明确异常——例如总奖品数超过 N 时直接报错 `Total prizes exceeds N`，避免静默回退产生不可验证的边界情况。参见 [核心抽奖算法详解](核心抽奖算法详解.md) 中关于 collision resolution 失败时的防护逻辑。

[来源](src/lottery.js#L9-L12)

## 验证：防御机制的实际运用

理解了攻击模型之后就知道验证时要检查什么：

1. **Round 一致性**：用 deadline 重新计算 round，与页面展示的 round 对比
2. **Randomness 拉取**：从 drand HTTP API 获取该 round 的 randomness，与页面展示的对比
3. **中奖结果重算**：用 randomness、N、prizeTiers 作为输入，自行执行 computeWinners，对比输出

这三步对应了 5 种攻击方式中第 1、4、5 的技术防御。第 2 和第 3 种攻击则通过检查 URL/短码中的参数值来验证。完整验证流程见 [粉丝：验证抽奖结果](粉丝：验证抽奖结果.md)。

[来源](ALGORITHM.md#L83-L98)

## 信任模型总结

```
┌────────────────────────────────────────────────────────┐
│                  整个抽奖系统的信任层                     │
├────────────────────────────────────────────────────────┤
│  技术层（无需信任）                                      │
│  ├─ drand 提供不可预测的 randomness  ← 数学保证         │
│  ├─ URL/短码锁定 deadline, N, prizes ← 编码保证         │
│  ├─ computeWinners 是纯确定性算法      ← 代码保证       │
│  └─ 排序规则硬编码声明                  ← 文档保证       │
├────────────────────────────────────────────────────────┤
│  社会层（仍需诚信）                                      │
│  ├─ deadline 前公开链接                ← 博主诚信       │
│  ├─ 公布完整候选列表                   ← 博主诚信       │
│  └─ 候选列表去重                       ← 博主诚信       │
└────────────────────────────────────────────────────────┘
```

这个模型的优雅之处在于：**技术层不需要任何信任，社会层只需要最低限度的诚信**。粉丝无需信任博主的人品，只需信任博主"至少在 deadline 前公开了链接"这一行为。
