# 短码设计与编解码

短码（short code）是 drand-draw 最具辨识度的设计之一——一条类似 `q-66364280-2s-1,3-2a,f,4e` 的字符串，将一次抽奖的全部参数压缩进不到 30 个字符。它不需要 URL 编码，不会被社交平台的文本截断破坏，可以原样粘贴在任何地方。

## 短码的五段结构

短码由连字符 `-` 分隔的 **至多五个段** 组成，前三个段是必选的，后两个段是可选的：

```
{chain}{deadline_hex}{N_base36}-{prizes_base36}-{winners_base36}
 ^          ^           ^          ^             ^
 |          |           |          |             └── 段 5: 中奖索引（可选）
 |          |           |          └── 段 4: 奖项层级（可选）
 |          |           └── 段 3: 参与者总数 N，base36
 |          └── 段 2: 截止时间戳，小端十六进制
 └── 段 1: 链前缀，单个字母
```

例如 `q-66364280-2s-1,3-2a,f,4e`：

| 段 | 值 | 含义 |
|----|------|---------|
| 1 | `q` | quicknet 链 |
| 2 | `66364280` | 截止时间戳 `1714828160`（十六进制） |
| 3 | `2s` | N = `100`（base36） |
| 4 | `1,3` | 奖项 = `1,3`（两个层级，各 1 和 3 个名额） |
| 5 | `2a,f,4e` | 中奖者索引 = `42,15,78` |

编解码并不要求最后两个段同时存在。当短码只用于配置阶段（截止前分享）时，段 4 和段 5 都可以省略；开奖后再补上 winners 段，变成完整的短码。

## 段 1：链前缀 `q` / `d` / `e`

drand 网络目前有三条链可用，每条链分配一个字母作为短码前缀。

| 链名 | 短码前缀 | 周期 | 创世时间戳 |
|------|----------|------|------------|
| `quicknet` | `q` | 3s | 1692803367 |
| `default` | `d` | 30s | 1595431050 |
| `evmnet` | `e` | 3s | 1727521075 |

JavaScript 端通过两个映射对象完成双向查找：

```javascript
// src/chains.js
export const CHAIN_SHORT = { q: 'quicknet', d: 'default', e: 'evmnet' }
export const SHORT_CHAIN = { quicknet: 'q', default: 'd', evmnet: 'e' }
```

Python 端使用等价的字典：

```python
# cli/drand_draw/encode.py
CHAINS = {
    'quicknet': { 'prefix': 'q', ... },
    'default':  { 'prefix': 'd', ... },
    'evmnet':   { 'prefix': 'e', ... },
}
PREFIX_TO_CHAIN = {v['prefix']: k for k, v in CHAINS.items()}
```

编码时用 `SHORT_CHAIN[params.chain]` 查前缀，解码时用 `CHAIN_SHORT[parts[0]]` 查回链名。

## 段 2：Deadline 用小端十六进制

截止时间以 Unix 时间戳（秒）存储，编码时转为 **不补零的小端十六进制字符串**。

```javascript
// src/encode.js
parts.push(params.deadline.toString(16))
```

```python
# cli/drand_draw/encode.py
deadline_hex = format(deadline, 'x')
```

为什么选十六进制而非十进制？

- **紧凑**：十进制 `1714828160` 需要 10 个字符，转十六进制后 `66364280` 仅 8 个字符，节省 20%。
- **可读**：时间戳的十六进制不是乱码——`66364280` 是规整的 8 位，人眼能一眼识别出"这里是一个时间戳"。
- **互逆**：JavaScript 的 `parseInt(hex, 16)` 和 Python 的 `int(hex, 16)` 行为一致，不存在跨语言歧义。

解码时用 `parseInt(parts[1], 16)` 还原时间戳。如果解码结果为 `NaN`，函数返回 `null` 表示无效。

## 段 3：N 用 base36

参与者总数 N 的范围是 1~1,048,575（详见 [安全边界与参数限制](安全边界与参数限制.md)），如果直接用十进制编码，最大值 `1048575` 需要 7 位。**Base36** 使用 0-9a-z 共 36 个字符，可将 7 位十进制数压缩到 4 位 base36：

| N（十进制） | N（base36） | 字符数节省 |
|------------|-------------|-----------|
| 100 | `2s` | 60% |
| 1000 | `rs` | 33% |
| 100000 | `255s` | 25% |
| 1048575 | `mh9z` | 43% |

JavaScript 直接使用 `Number.prototype.toString(36)`：

```javascript
parts.push(params.n.toString(36))
```

Python 需要手动实现 base36（因为 Python 的 `int()` 支持 36 进制，但 `format(n, '36')` 不存在）：

```python
_BASE36_CHARS = '0123456789abcdefghijklmnopqrstuvwxyz'

def to_base36(n):
    if n == 0:
        return '0'
    result = []
    while n > 0:
        result.append(_BASE36_CHARS[n % 36])
        n //= 36
    return ''.join(reversed(result))
```

解码则统一使用 `int(s, 36)`。

## 段 4 和段 5：Prizes 和 Winners 的可选编码

段 4 存储奖项层级配置（如 `1,3` 代表一奖 1 人、二奖 3 人），段 5 存储中奖索引列表（如 `2a,f,4e` 代表索引 42、15、78）。二者都是 **逗号分隔的 base36 数字**。

```javascript
// src/encode.js — 编码
if (params.prizes && params.prizes.length) {
  parts.push(params.prizes.map(p => p.toString(36)).join(','))
}
if (params.winners && params.winners.length) {
  parts.push(params.winners.map(w => w.toString(36)).join(','))
}

// src/encode.js — 解码
if (parts[3]) {
  result.prizes = parts[3].split(',').map(x => parseInt(x, 36)).filter(x => !isNaN(x))
}
if (parts[4]) {
  result.winners = parts[4].split(',').map(x => parseInt(x, 36)).filter(x => !isNaN(x))
}
```

Python 端使用同样的逻辑，但增加了基本的格式校验：`decode_short_code` 在段数小于 3 或前缀不匹配时抛出 `ValueError`，而非静默返回 `null`。这是两套实现之间唯一的语义差异。

## 编解码的互逆性

短码设计保证编码和解码互为逆操作——对任意合法参数反复编解码，始终得到一致的语义。以下验证用例覆盖了所有段组合：

| 输入参数 | 编码结果 | 解码后验证 |
|----------|----------|-----------|
| `{chain:"quicknet", deadline:1714828160, n:100, prizes:[1,3]}` | `q-66364280-2s-1,3` | chain=quicknet, deadline=1714828160, n=100, prizes=[1,3], winners=[] |
| `{chain:"quicknet", deadline:1714828160, n:100, prizes:[1,3], winners:[42,15,78,33]}` | `q-66364280-2s-1,3-2a,f,4e,21` | winners 还原为 [42,15,78,33] |
| `{chain:"default", deadline:1715000000, n:50000, prizes:[5]}` | `d-663969c0-yo-5` | n=50000, base36 `yo` 解码为 50000 |
| `{chain:"evmnet", deadline:1728000000, n:1000000}` | `e-66ff4480-l0r` | 无 prizes/winners 时后两个段省略 |

在两个语言实现上做同样的编解码轮换，结果一致：

```bash
# Python CLI
$ python -m drand_draw encode --chain quicknet --deadline 1714828160 --n 100 --prizes 1,3 --winners 42,15,78,33
q-66364280-2s-1,3-2a,f,4e,21

$ python -m drand_draw decode q-66364280-2s-1,3-2a,f,4e,21
Chain: quicknet
Deadline: 1714828160
N: 100
Prizes: 1, 3
Winners: #42, #15, #78, #33
```

```javascript
// JavaScript 浏览器控制台
> encodeShortCode({chain:"quicknet", deadline:1714828160, n:100, prizes:[1,3], winners:[42,15,78,33]})
"q-66364280-2s-1,3-2a,f,4e,21"

> decodeShortCode("q-66364280-2s-1,3-2a,f,4e,21")
{chain:"quicknet", deadline:1714828160, n:100, prizes:[1,3], winners:[42,15,78,33]}
```

## 与 URL 格式的关系

系统同时支持 URL 格式（`#/?chain=quicknet&deadline=1714828160&n=100&prizes=1,3`）和短码格式。二者的转换由 `hashToParams` 和 `paramsToHash` 协调：输入的 hash 如果是 `?key=value` 格式就走 URL 解码，如果是字母-数字-连字符格式则走短码解码。同一个抽奖参数可以两种格式互相转换，信息不丢失。

详细设计见 [理解 URL 与短码格式](理解-url-与短码格式.md)。
