整个抽奖系统的技术核心只有两个数学函数：**确定从哪一轮 drand 信标获取随机数**，以及 **将随机数转化为一组中奖编号**。理解这两个函数，就理解了整个系统的不可作弊性从何而来。

---

## computeRound：锁定 drand 信标轮次

抽奖的"不可预测性"来源不是算法本身——算法完全公开——而是 drand 网络在截止时间之前无法预知的随机数。`computeRound` 的任务就是把博主设定的截止时间，映射到 drand 链上的一个确定轮次。

### 公式

```
round = floor((deadline - genesis) / period) + 1
```

四个参数的含义：

| 参数 | 来源 | 说明 |
|------|------|------|
| `deadline` | 博主设定 | 抽奖截止的 Unix 时间戳（秒） |
| `genesis` | 链参数 | drand 链的创世时间戳，**硬编码** 在 `src/chains.js` 中 |
| `period` | 链参数 | 该链生成随机数的间隔秒数，同样 **硬编码** |
| `+1` | 约定 | drand round 编号从 1 开始，而非 0 |

[来源](src/lottery.js#L3-L8) | [来源](cli/drand_draw/lottery.py#L4-L9)

### JavaScript / Python 双实现

```js
// src/lottery.js
export function computeRound(chainId, deadlineUnix) {
  const chain = CHAINS[chainId]
  const elapsed = deadlineUnix - chain.genesisTime
  if (elapsed < 0) return -1
  return Math.floor(elapsed / chain.period) + 1
}
```

```python
# cli/drand_draw/lottery.py
def compute_round(deadline, genesis, period):
    elapsed = deadline - genesis
    if elapsed < 0:
        return -1
    return math.floor(elapsed / period) + 1
```

两段代码的差异仅在参数传递方式：JavaScript 版接收 `chainId` 后在函数内查表，Python 版直接接收三个数值。**计算结果完全一致。** 当 `deadline < genesis` 时返回 `-1`，这是唯一的异常情况。

[来源](src/lottery.js#L3-L8) | [来源](cli/drand_draw/lottery.py#L4-L9)

### 数值推演

以 quicknet 链为例（`genesis = 1692803367`，`period = 3`），假设博主设定 `deadline = 1692803400`：

```
elapsed = 1692803400 - 1692803367 = 33
round   = floor(33 / 3) + 1 = 11 + 1 = 12
```

drand 网络会在第 12 轮发布随机数，任何人无法提前预知。

## computeWinners：随机数 → 中奖编号

拿到 randomness（一个 64 字符的十六进制字符串）之后，需要将其转化为一组 `[0, N-1]` 区间内的编号。`computeWinners` 的算法设计遵循三个原则：

1. **确定性** —— 同样的 randomness + N + prizeTiers 永远输出同样的 winners
2. **不可预测性** —— 每个编号的 seed 都融合了 randomness 的不同片段
3. **无偏性** —— 取模运算 + 环形碰撞处理，保证每个编号中奖概率均等

[来源](src/lottery.js#L10-L33) | [来源](cli/drand_draw/lottery.py#L11-L33)

### 种子拼接：randomness + shift

```js
const shift = winners.length
const seedHex = randomness + shift.toString(16)
const bigVal = BigInt('0x' + seedHex)
```

`shift` 等于 **当前已产生的赢家数量**。第一个奖位 `shift = 0`，第二个 `shift = 1`，以此类推。seed 是 **randomness 字符串后直接追加 shift 的十六进制形式**（无 `0x` 前缀），然后整体作为十六进制字符串解析为 `BigInt`。

这样设计的原因：如果直接对 randomness 取模 N，所有奖位会算出相同的 idx，失去多样性。通过拼接不同的 shift 值，每个奖位都使用了 randomness 的不同"分片"。

[来源](src/lottery.js#L16-L18) | [来源](cli/drand_draw/lottery.py#L19-L21)

### BigInt 取模

```js
let idx = Number(bigVal % BigInt(n))
```

对 `n` 取模将结果映射到 `[0, n-1]` 区间。`BigInt` 的使用是必要的——seedHex 的长度是 64 + shift 的十六进制位数，远超 JavaScript 原生 Number 的安全整数范围（2^53）。Python 的 `int` 原生支持任意精度，因此无需特殊处理。

[来源](src/lottery.js#L19) | [来源](cli/drand_draw/lottery.py#L22)

### 碰撞顺延：环形查找

```js
let attempts = 0
while (used.has(idx)) {
  idx = (idx + 1) % n
  attempts++
  if (attempts >= n) throw new Error('Collision resolution failed')
}
used.add(idx)
winners.push(idx)
```

当两个奖位算出相同的 `idx` 时——理论上概率为 `1/n`——算法不会跳过冲突的奖位，而是 **顺延到下一个未被占用的编号**，采用环形查找（`(idx + 1) % n`）。这意味着：

- 只要 `总奖数 ≤ N`，一定能找到空位
- `attempts >= n` 的保护检查是防御性编程，正常情况下不会触发
- **不会改变中奖总人数**，也不会跳过某个奖项层级

[来源](src/lottery.js#L20-L27) | [来源](cli/drand_draw/lottery.py#L23-L30)

### formatWinnerList：分组输出

`computeWinners` 返回一维数组，`formatWinnerList` 按 `prizeTiers` 切分：

```js
// prizes = [1, 3], winners = [42, 17, 88, 5]
// 返回 [[42], [17, 88, 5]]
```

## 手动演算：从头到尾算一遍

用一组示例数据完整走一遍计算流程。

### 输入参数

```
randomness = "a3f25c8d1e7b9f0846c2a5d3e8f1b7c4a9d0e6f3b8c1a4d7e0f9b2c5a8d1e4f7b"
             （64 字符十六进制）
N          = 100
prizeTiers = [1, 3]   → 共 4 个奖位
```

### 第 1 步：shift = 0（第一个一等奖）

```
seedHex = randomness + "0"
        = "a3f25c8d...e4f7b0"
bigVal  = BigInt(seedHex)         → 一个极大的整数
idx     = bigVal % 100            → 假设结果为 42
used    = {42}
winners = [42]
```

### 第 2 步：shift = 1（第一个二等奖）

```
seedHex = randomness + "1"
        = "a3f25c8d...e4f7b1"
bigVal  = BigInt(seedHex)
idx     = bigVal % 100            → 假设结果为 17
17 不在 used 中，无碰撞
used    = {42, 17}
winners = [42, 17]
```

### 第 3 步：shift = 2（第二个二等奖）

```
seedHex = randomness + "2"
idx     = bigVal % 100            → 假设结果为 17（碰撞！）
17 已在 used 中 → idx = (17 + 1) % 100 = 18
18 未被占用
used    = {42, 17, 18}
winners = [42, 17, 18]
```

### 第 4 步：shift = 3（第三个二等奖）

```
seedHex = randomness + "3"
idx     = bigVal % 100            → 假设结果为 88
88 未被占用
used    = {42, 17, 18, 88}
winners = [42, 17, 18, 88]
```

### 输出

```
formatWinnerList([42, 17, 18, 88], [1, 3])
→ [[42], [17, 18, 88]]
```

一等奖：编号 42；二等奖：编号 17、18、88。

## 双语言一致性与交叉验证

JavaScript 和 Python 的实现在语义上 **逐行等价**：

| 步骤 | JavaScript | Python |
|------|-----------|--------|
| 种子拼接 | `randomness + shift.toString(16)` | `randomness + format(shift, 'x')` |
| 大整数解析 | `BigInt('0x' + seedHex)` | `int(seedHex, 16)` |
| 取模 | `bigVal % BigInt(n)` | `big_val % n` |
| 碰撞检测 | `used.has(idx)` | `idx in used` |
| 环形推进 | `(idx + 1) % n` | `(idx + 1) % n` |
| 异常保护 | `if (attempts >= n)` | `if attempts >= n` |

验证思路非常简单：**用同一组输入分别调用两套实现，比对输出。** 在 CI/CD 流程中，可以增加一个 cross-check 步骤：

```bash
# 用 Python CLI 验证 JavaScript 计算结果
python3 -m drand_draw compute \
  --chain quicknet \
  --deadline 1692803400 \
  --n 100 \
  --prizes 1,3
```

然后对比输出的 winners 数组是否与 Web 端展示的完全一致。

## 安全边界

算法层面的安全边界在 [ALGORITHM.md](ALGORITHM.md#7-安全边界) 中有完整定义：

| 参数 | 最大值 | 技术原因 |
|------|--------|----------|
| N | 1,048,575 | BigInt 取模运算的合理上限 |
| 奖项层级 | 15 层 | 避免 seed 碰撞链过长 |
| 每层人数 | 1,000 | 防止 collisions 退化到 O(n²) |
| deadline | 2106 年 | Unix 32 位时间戳上限 |

超出这些边界后，虽然算法在数学上仍然正确，但实际运行可能遇到性能问题或边界条件。详见 [安全边界与参数限制](安全边界与参数限制.md)。
